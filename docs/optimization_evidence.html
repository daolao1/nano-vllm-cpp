nano-vllm-cpp · Decode 优化 Profiling 证据

核心发现：两个阶段需要两套优化

Prefill（rows ≈ 数百到上千 token）是 compute-bound：attention 占 80%+ 时间，QKT 和 PV 是 O(N²) 矩阵乘。优化方向 = 用 Flash Attention 替换手写 attention。实测 1041 → 198 ms（−81%）。

Decode（每步只 1 个新 token，rows = batch_size ≈ 20）是 launch-bound：每个小 kernel 只干 5,376 字节的活，但一次 kernel launch 就要 5–8 µs。优化方向 = 融合 kernel 减少 launch 次数 + CUDA Graph 消除剩余 launch overhead。实测 22.1 → 11.2 ms（−49%）。

⚠ 关键认知：用「prefill 的优化思路」（榨 HBM 带宽）去做 decode kernel 是徒劳的——decode 的 kernel 单独看带宽利用率只有 0.3%，但延迟全是 launch overhead，再怎么榨带宽都不会变快。反过来用「decode 的优化思路」（融合）去做 prefill 也没用——prefill kernel 已经在 memory-bound 区饱和。

1. 方法论与 profiling 工具选型

1.1 期望工具

标准方法论是 Nsight Compute (ncu) + Nsight Systems (nsys)：

  • ncu --metrics dram__bytes,sm__warps_active 拿单 kernel 的 HBM 带宽、SM 利用率、roofline 位置
  • nsys profile --trace=cuda 拿端到端时间线，定位 launch gap 和 stream stalls

1.2 实际遇到的问题

$ ncu --metrics gpu__time_duration.sum ./build/bench_e2e ...
==ERROR== ERR_NVGPUCTRPERM - The user does not have permission to access
NVIDIA GPU Performance Counters on the target device 0.
$ sudo ncu ...
==ERROR== 同上（即便 root 也不行）

原因：WSL2 上 GPU 驱动是 Windows 侧的，Linux 侧 NVreg_RestrictProfilingToAdminUsers 内核参数无效， 必须在 Windows 注册表改 HKLM\SYSTEM\...\NVIDIA Display.Driver\NVPCS，且需要重启。这是 NVIDIA 官方记录的 WSL2 已知限制。

同样地，nsys 2022.4 在这个 WSL 内核版本下抓不到 CUDA kernel runtime 事件，只能拿到 API trace（102 events 全是 cudaMalloc/cudaMemcpy）。

1.3 替代方案：cudaEvent_t 微基准

这是 ncu 不可用时的业界标准替代——CUDA Runtime API 提供的高精度计时器，无需驱动权限：

cudaEvent_t start, stop;
cudaEventCreate(&start); cudaEventCreate(&stop);
cudaEventRecord(start, stream);
fn();                           // launch kernel
cudaEventRecord(stop, stream);
cudaEventSynchronize(stop);
float ms;
cudaEventElapsedTime(&ms, start, stop);

能拿到的：每 kernel 中位数延迟、推算 HBM 有效带宽、batch sweep 区分 launch-bound vs memory-bound。 拿不到的：SM 利用率、寄存器压力、L2 hit rate。但对当前问题（确认 decode 是 launch-bound）已经够用。

Bench 代码：bench/bench_kernels.cpp（200 行，与项目同栈编译，复用真实 kernel 入口）。

3. Prefill vs Decode：两个完全不同的战场

这是本页最核心的一节。把两个阶段的工作负载特征摆在一起对比，所有后续数据都围绕这张表展开。

术语：rows 是什么？

本页所有 kernel 的输入张量 shape 都是 [rows, hidden]（hidden = 896，bf16）。rows = 一次 kernel launch 要处理的 token 行数，每个 kernel 都是按 token 一行行处理。具体在三个场景下的含义：

  • decode：每个序列每步只产生 1 个新 token，所以 rows = batch_size（bench 实测 avg batch ≈ 19.3，所以 decode 的实际 rows ≈ 20）
  • prefill：一次喂整段 prompt，所以 rows = 这个 batch 里所有 prompt token 数的总和（实测每步几百到上千）
  • micro-bench（§5）：我在 sweep 里手动取 rows ∈ {1, 4, 16, 64, 256, 1024}，跨越 launch-bound → memory-bound 的过渡，从而能定位每个工作点

例：rms_norm rows=20 时输入约 36 KB，rows=1024 时输入约 1.8 MB——两者差 50 倍但落在同一 kernel 的两端，正好对应 decode 和 prefill 工作点。

3.2 Prefill：compute-bound，FA 主导

Prefill 一次喂进上百 token，每个 token 的 hidden 张量直接把所有 kernel 推到 sweep 表的 rows ≥ 256 区。在这一区：

  • kernel 实际工作时间 >> launch overhead（add_rms_norm @ 1024 已经触顶 314 GB/s，见 §6）
  • 融合 kernel 在 prefill 区仍然更快约 2×（实测见下方对比表），但绝对节省被 attention 稀释成总时长的 < 1%
  • 真正的瓶颈是 attention 的 O(N²) 计算，QKT 是 dense GEMM，PV 也是 dense GEMM，total compute 比所有 elementwise kernel 加起来还多一个数量级

所以 prefill 优化 = 把 attention 换成 Flash Attention。FA 的核心贡献是用 SRAM tiling 把 PV 和 softmax 融合在 register 里，避免把 N×N 的 attention matrix 写回 HBM。这是「换算法」，不是「调 kernel」。

实测：fused vs separate 在 sweep 全区间

在 bench_kernels.cpp 里加了一项 SEPARATE 5-kernel chain：把 split_qkv → q_norm → k_norm → rotary → store_kvcache 这 5 步串起来计时，跟 fused kernel 同输入对比。

rows阶段fused (µs)separate 5-kernel (µs)speedup
1decode batch=15.1234.826.8×
16decode 工作点6.1435.845.8×
64decode 高 batch8.1940.675.0×
256短 prompt prefill16.1041.982.6×
1024prefill 工作点49.15103.422.1×

反直觉：fused 在 prefill 区也快 2×。原因不是带宽，是固定开销——separate 链路有 5 次 launch × ~6 µs = 30 µs 的 launch overhead，即使 rows=1024 也占 30%；加上中间张量（q、k、normalized_q、normalized_k）的 HBM 往返流量。两项加起来让 fused 在所有 rows 都赢。

那为什么 prefill 不复用 fused？查代码 src/models/transformer.cpp:127：!ctx.is_prefill 直接排除了 prefill。每层 prefill 省 54 µs × 28 层 = 1.5 ms/step，占 198 ms prefill 的 0.76%——绝对值太小，加上 prefill 的 attention 走 FA varlen（用的是 attn_.forward 而非 forward_decode_flash_only），改造代码风险 vs 收益不划算。这是工程权衡，不是性能权衡。

实测端到端：FA 不可用时 prefill = 1040.7 ms/step；FA 加载后 prefill = 258 ms/step（eager）/ 198 ms/step（graph）。5.2× 提速完全来自 FA，融合 kernel 即便复用到 prefill 也只能再省 0.76%。

3.3 Decode：launch-bound，融合+Graph 主导

Decode 每步只产生 1 个新 token，整个 batch 的 rows = batch_size。bench 里 avg batch = 19.3，所以所有 kernel 都跑在 sweep 表的 rows ≤ 64 区。在这一区：

  • 实测 rms_norm @ rows=20 ≈ 6.14 µs，但只搬 ~110 KB → 有效带宽 ~18 GB/s（峰值的 7%）
  • 这 6 µs 不是访存时间，是 cudaLaunchKernel 的 host driver + GPU front-end 调度开销。证据：rows=1 和 rows=64 延迟都是 6.14 µs（bytes 涨了 40 倍延迟却不变）
  • 每个 transformer 层有 ~10 个 kernel × 28 层 = 280 次 launch × 6 µs = 1.68 ms 是 launch overhead，占 11.2 ms decode 步时长的 15%

所以 decode 优化 = 减少 kernel 数 + 消除 launch overhead，两条手段：

  1. 5-kernel 融合（split_qkv + q_norm + k_norm + rotary + store_kvcache → 1 个）：每层省 4 次 launch ≈ 24.7 µs，28 层省 692 µs/step
  2. CUDA Graph capture：把整个 decode step 录成一个 graph，replay 时跳过几百次 cudaLaunchKernel，每步再省 ~0.4 ms

实测：融合 kernel 单独 ≈ 6.14 µs vs 5 个独立 kernel ≈ 30.87 µs（micro-bench）。端到端：FA+融合 eager = 11.6 ms，加 graph = 11.2 ms（再省 0.4 ms）。两条优化叠加：22.1 → 11.2 ms = 2.0× 提速。

5.2 Launch-bound 诊断（仅适用于 decode）

关键判据：对每个 kernel，从 rows=1 到 rows=64，bytes 涨了 ~40×，但延迟几乎不变（5-8 µs）。 这意味着计算和访存时间被 launch overhead 完全掩盖，kernel 实际「干活」的部分 < 1 µs。

  • 5-8 µs 是什么：这正是 NVIDIA RTX 40 系列上一次 cudaLaunchKernel 的端到端开销（host driver 调度 + GPU front-end 解码 + warp 启动）
  • 什么时候不再是 launch-bound：rows ≥ 256 后延迟开始随 bytes 线性增长 → 进入 memory-bound 区
  • Decode 的实际工作点：bench 里 avg batch = 19.3，落在 launch-bound 区的中段 → 融合 kernel 是正确的优化方向

5.3 5-kernel 融合的实际收益（decode 工作点 rows≈20）

设计权衡：融合 kernel 单看带宽利用率（38%）不如分离 kernel（rms_norm 66%, split_qkv 94%）。 但 decode 工作负载是 launch-bound 区，单 kernel 带宽饱和度不重要，kernel 总数才重要。 融合的本质是把「5 个高带宽利用率的 kernel」换成「1 个中带宽利用率的 kernel」，减少 4 次 launch overhead。

7. Flash Attention 替换的证据（Prefill 主战场）

Prefill 路径从 1041 ms → 198 ms 的提速，主因是用官方 FA varlen 替代手写 prefill_paged_attention 兜底。两组实测：

路径Prefill ms/stepDecode ms/stepThroughput

注意第二行的 decode（11.6 ms）相比第一行（22.1 ms）也快了一半，因为 decode attention 也走 FA splitkv 路径。 所以 FA 同时主导 prefill 和 decode 提速，融合 + Graph 是在此基础上的 ~5% 边际优化。

8. store_kvcache 向量化的诚实评估

改动

新增 store_kvcache_vec_kernel：用 int4 一次搬 16 字节（bf16/fp16 = 8 元素，fp32 = 4 元素）； 保留原 kernel 作为 fallback，由 can_vectorize_store_kvcache_16B 守卫（row_bytes % 16 == 0 + 4 个指针 16-byte aligned）。

实测对比

store_kvcache 在 micro-bench 里达到 102.4 GB/s @ rows=1024（37.6% peak），改动前是逐元素拷贝（estimated < 60 GB/s）。 但是：decode 主路径已经被 fused_split_norm_rope_store 取代，不调用 store_kvcache。

诚实评估

这次优化的真实价值：

  • decode bf16 主路径：无观测收益（被融合 kernel 绕过）
  • decode int8 KV cache 路径：有收益但未实测（int8 路径不能融合，仍走 store_kvcache，但量化版本是 store_kvcache_int8 不是这个）
  • prefill 路径：理论收益 ~10 µs/层，但 prefill 是 compute-bound（FA 占 80%+），整体观测不到
  • 真正的价值：给未来的 INT4 / FP8 KV cache 路径预留性能基线（那些路径会重新走 store_kvcache）

—— 端到端 bench 显示改动前后吞吐都在 1611 tok/s 左右（噪声内）。这是符合方法论第 4 步「< 5% 收益且简化代码 → 提交」的边际优化。

  • 没有 ncu 数据：WSL2 限制，下面的指标我没法直接给——SM 利用率、L2 hit rate、warp stall 类型分布、roofline 的精确 AI 坐标。本页的 roofline 推断基于 cudaEvent 时间 + 已知 byte 量，是合理推断不是直接测量。在原生 Linux + 解锁 perfcounter 的环境下，所有 ncu 数据都能立即补齐。
  • 没有 nsys 时间线图：同样原因，nsys 在这个 WSL 内核版本下抓不到 kernel runtime 事件。我能给出 cudaEvent 计时但无法给出可视化时间线。
  • 5-kernel 融合的 692 µs/step 节省是估算：基于 micro-bench 单 kernel 时间 × 28 层。实测端到端差是 0.3-0.4 ms（cuda graph 与 eager 的差），数量级一致但不完全等同——剩余差距应该来自 CUDA Graph 本身的 launch 消除和 ScratchAllocator。
  • HBM 峰值 272 GB/s 是手册值：实测 add_rms_norm 触顶 314.7 GB/s 说明物理峰值更高。本页的 %peak 列偏保守，但 launch-bound 的判断不受影响。
  • 所有数据是单次复现，不是平均多次：cudaEvent 内部已经做了 200 次中位数稳定化，但端到端 bench_e2e 的方差没量化（粗估 ±2%）。

最终结论：本页给出的不是「ncu 黑盒数字证明 X 是 Y 倍快」式的精确报告，而是「用 cudaEvent 微基准证明 decode 是 launch-bound，所以融合是正确方向；用端到端壁钟证明融合 + Graph 实际省 0.4 ms/step；两者数量级互相印证」的逻辑闭环。
