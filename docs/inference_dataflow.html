1. 推理路径总览

一句话总结：这条链路不是“抽象 Transformer 教材图”，而是当前 nano-vllm-cpp 已经实现并可运行的 Qwen3 前向路径。模型层负责算子组合，Context 负责把 Prefill / Decode、KV Cache、序列边界和页表信息灌进运行时。

模型入口

Qwen3ForCausalLM::forward 接收 input_ids 和 positions，先走 model_.forward，最后交给 lm_head_.forward 产出 logits。

控制入口

Prefill 还是 Decode 不是模型自己推断，而是调用方提前写入 Context。Attention、LM Head 和页式 KV Cache 都依赖这些元数据。

算子主干

整条主干是 Embedding - DecoderLayer x L - Final RMSNorm - LM Head。每个 DecoderLayer 内部再拆成 input_layernorm、self_attn、post_attention_layernorm、mlp。

并行语义

TP 不是只切权重。Embedding、RowParallelLinear、LM Head 都会触发通信；QKV 和 gate/up 则保持本地 shard 计算，等后续阶段再汇合。

input_ids [N] + positions [N] + Context │ ▼ Qwen3ForCausalLM::forward │ ├── Qwen3Model::forward │ ├── VocabParallelEmbedding -> hidden_states [N, H] │ ├── for each decoder layer: │ │ input_layernorm │ │ qkv_proj -> q/k/v -> q_norm/k_norm -> RoPE │ │ Attention(q, k, v, Context, KV Cache) │ │ o_proj │ │ post_attention_layernorm │ │ gate_up_proj -> SiluAndMul -> down_proj │ └── final RMSNorm -> hidden [N, H] or [rows, H] │ └── ParallelLMHead ├── prefill 时可先 gather_last_tokens ├── local logits [rows, vocab_local] └── TP 时 gather_last_dim_to_rank0
模型只组合算子 Context 决定执行分支 Attention 负责读写 KV Cache TP 通信嵌入真实 forward

2. 输入、Context 与控制面 Context / KV Cache

虽然模型前向签名很短，但真正决定推理行为的元数据大多不在函数参数里，而在 thread_local 的 Context 中。理解这一层，才能看清 Prefill、Decode 和页式缓存是怎样进入算子的。

字段 形状或语义 谁来消费 影响
input_ids 通常是扁平 token id，形状 [N] VocabParallelEmbedding 查词表权重，产生 [N, H] 的 hidden_states。
positions 形状 [N] RotaryEmbedding 对 q 和 k 做位置旋转。
is_prefill true 表示 Prefill，false 表示 Decode Attention, ParallelLMHead 决定走 dense prefill、paged prefill 还是 decode paged attention，也决定 lm head 是否只保留每条序列最后一个 token。
cu_seqlens_q, cu_seqlens_k prefix sum 形式的序列边界 Attention, ParallelLMHead Prefill attention 需要它划分序列，lm head 的 gather_last_tokens 也依赖它定位每条序列的尾 token。
slot_mapping 逻辑 token 到 KV cache 物理槽位的映射 Attention 在真正读 attention 之前，先把当前步的 key/value 写到对应 cache 页。
context_lens 每条序列当前上下文长度 Decode attention 告诉 decode kernel 每个 query 需要回看多长的历史。
block_tables 逻辑 block 到物理 block 的页表 PagedAttention 决定注意力如何通过页表访问缓存中的历史 K/V。
当前实现里 Context 存在于 src/utils/context.cpp 的 thread_local 存储中。这意味着多线程、多 rank 前向可以并发持有不同的页表、序列边界和 slot mapping，而不会互相覆盖。
模型不会自动猜测“现在是 prefill 还是 decode”。如果调用方没有先设置正确的 Context，那么 forward 虽然能进到模型代码，但 attention 分支、KV 写入和 lm head 行为都会失真。

3. 端到端组合图

下面这张图对应的是当前 src/models/qwen3.cpp 里真正的 forward 顺序。这里的 N 是本轮参与计算的 token 数，H 是 hidden_size，Hq 是本 rank 的 query 头数乘以 head_dim，Hkv 是本 rank 的 kv 头数乘以 head_dim，I 是本 rank 的 intermediate_size。

input_ids [N] + positions [N] + Context | v VocabParallelEmbedding local lookup(weight shard[vocab_local, H]) -> all_reduce_sum if tp_size > 1 -> hidden_states [N, H] | v for each decoder layer: if residual undefined: residual = hidden_states hidden = RMSNorm(hidden_states) else: hidden, residual = AddRMSNorm(hidden_states, residual) qkv = QKVParallelLinear(hidden) -> [N, Hq + Hkv + Hkv] q = reshape(qkv[..., :Hq]) -> [N, local_q_heads, head_dim] k = reshape(qkv[..., Hq:Hq+Hkv]) -> [N, local_kv_heads, head_dim] v = reshape(qkv[..., Hq+Hkv:]) -> [N, local_kv_heads, head_dim] q = q_norm(q), k = k_norm(k) -> when attention_bias == false rotary_emb(positions, q, k) -> in-place RoPE attn_out = Attention(q, k, v, Context, KV Cache) -> [N, local_q_heads, head_dim] attn_out = reshape(attn_out) -> [N, Hq] attn_out = RowParallelLinear(attn_out) -> [N, H], all_reduce_sum if tp_size > 1 hidden, residual = AddRMSNorm(attn_out, residual) gate_up = MergedColumnParallelLinear(hidden) -> [N, 2I] mlp_out = SiluAndMul(gate_up) -> [N, I] hidden_states = RowParallelLinear(mlp_out) -> [N, H], all_reduce_sum if tp_size > 1 after last layer: hidden = RMSNorm(hidden_states + residual) ParallelLMHead: if prefill and cu_seqlens_q defined: hidden = gather_last_tokens(hidden) -> [num_seqs, H] local_logits = hidden @ weight_local^T -> [rows, vocab_local] logits = gather_last_dim_to_rank0 if tp_size > 1 -> rank0:[rows, vocab], other ranks: undefined
这里最容易误解的点是 residual 流。代码里不是每个子层都显式写一条 residual add，而是通过 RMSNorm::forward(input, residual) 的融合路径，把 residual 累加和归一化合并执行，所以 layer.forward 返回的是一对张量，而不是单个 hidden。

4. Embedding 与 residual 起点

模型入口是 Qwen3Model::forward。第一步不是线性层，而是 VocabParallelEmbedding::forward。它的输出一方面成为第一层的 hidden_states，另一方面也成为第一层 residual 的起点。

VocabParallelEmbedding 做什么

TP=1 时它就是普通 embedding lookup。TP>1 时，每个 rank 只持有自己的词表分片 [vocab_local, H]，本 rank 只对本地词表命中的 token 产出非零 embedding，之后通过 all_reduce_sum 合成完整 [N, H] hidden_states。

首层 residual 为什么不是 0

第一层开始时 residual 尚未定义，所以代码会先把 residual = hidden_states，再只对当前 hidden_states 做一次 RMSNorm。这样 attention 之后，post_attention_layernorm 才能把 embedding 分支加回主干。

Tensor hidden_states = embed_tokens_.forward(input_ids, allocator, stream);
Tensor residual;

for (Qwen3DecoderLayer& layer : layers_) {
    auto layer_output = layer.forward(positions, hidden_states, residual, allocator, stream);
    hidden_states = layer_output.first;
    residual = layer_output.second;
}
输入: [N] 输出: [N, H] TP>1 时 embedding 做 all_reduce_sum 首层 residual 起点是 embedding 输出

5. DecoderLayer 内部数据流

一个 Qwen3DecoderLayer 由 input_layernorm、self_attn、post_attention_layernorm、mlp 四个状态组件组成，但它真正编码的是两段“先把 residual 累加起来，再做归一化”的 PreNorm 残差流。

5.1 进 Attention 前

如果 residual 还没建立: next_residual = hidden_states hidden = input_layernorm(hidden_states) 如果 residual 已存在: hidden, next_residual = input_layernorm(hidden_states, residual) 等价于: next_residual = hidden_states + residual hidden = RMSNorm(next_residual)

5.2 Attention 后进 MLP 前

attn_out = self_attn(hidden) post_attn_hidden, next_residual = post_attention_layernorm(attn_out, next_residual) hidden = mlp(post_attn_hidden) return {hidden, next_residual}
if (!next_residual.defined()) {
    next_residual = hidden_states;
    hidden = input_layernorm_.forward(hidden_states, allocator, stream);
} else {
    auto norm_pair = input_layernorm_.forward(hidden_states, next_residual, allocator, stream);
    hidden = norm_pair.first;
    next_residual = norm_pair.second;
}

hidden = self_attn_.forward(positions, hidden, allocator, stream);
auto post_attn = post_attention_layernorm_.forward(hidden, next_residual, allocator, stream);
hidden = mlp_.forward(post_attn.first, allocator, stream);
return {hidden, post_attn.second};
这里的 hidden 更像“当前子层新算出来、尚未并入 residual 的分支输出”；residual 则表示到当前时刻为止已经累计好的主干和。最后一层结束后，模型还会再做一次 norm_.forward(hidden, residual) 收口。

6. Attention 的计算与分支 dense / pageddecode

Qwen3Attention 是整条链路里张量变形最多的地方。它先做 QKV 投影，再切片重排，再做 q_norm、k_norm 和 RoPE，最后进入 attention kernel，并通过 o_proj 投回 hidden_size。

QKVParallelLinear

输入是 [N, H]，输出是 [N, Hq + Hkv + Hkv]。TP>1 时，Q 头和 KV 头都按输出维切 shard，每个 rank 只负责自己的 q/k/v 头。

slice + reshape

Q、K、V 不是通过三条独立线性层得到，而是从一个 qkv 大张量沿最后一维切出，再 reshape 成 [N, heads, head_dim]。

RoPE 前归一化

当前 Qwen3Attention 在 attention_bias == false 的路径下会创建 q_norm 和 k_norm，因此 Qwen3 默认路径是先做 RMSNorm，再做 RoPE。

Tensor qkv = qkv_proj_.forward(hidden_states, allocator, stream);
Tensor q = slice_last_dim_contiguous(qkv, 0, q_size_, allocator, stream)
             .reshape({rows, num_heads_, head_dim_});
Tensor k = slice_last_dim_contiguous(qkv, q_size_, kv_size_, allocator, stream)
             .reshape({rows, num_kv_heads_, head_dim_});
Tensor v = slice_last_dim_contiguous(qkv, q_size_ + kv_size_, kv_size_, allocator, stream)
             .reshape({rows, num_kv_heads_, head_dim_});

q = q_norm_->forward(q, allocator, stream);
k = k_norm_->forward(k, allocator, stream);
rotary_emb_.forward(positions, q, k, stream);

Tensor output = attn_.forward(q, k, v, allocator, stream);
return o_proj_.forward(output.reshape({rows, q_size_}), allocator, stream);

6.1 Attention::forward 的三条执行路

分支 触发条件 核心行为
Prefill dense is_prefill == true 且没有 block_tables 不读 KV Cache，直接用当前步传入的 key/value 做 dense prefill，入口是 launch_prefill_attention_dense。
Prefill paged is_prefill == true 且存在 cache 与 block_tables 若有 slot_mapping 先写 KV，再用 block_tables 和 cu_seqlens_k 从页式缓存回看上下文，入口是 launch_prefill_attention_paged。
Decode paged is_prefill == false 要求同时具备 k_cache、v_cache、context_lens、block_tables，先写本步 KV，再走 launch_decode_attention_paged 回看整段历史。

6.2 Attention 内部的阶段切分

阶段 关键张量 发生的事情
KV 写入前 query, key, value 检查 defined、CUDA、contiguous、float32、heads 数和 head_dim 是否匹配配置。
写 KV Cache slot_mapping, k_cache, v_cache 如果缓存已绑定且 slot_mapping 已定义，就调用 store_kvcache 把本步 K/V 写到页式缓存。
读注意力上下文 cu_seqlens_* 或 block_tables, context_lens 按 Prefill 或 Decode 分支选择 dense / paged kernel。
投回 hidden_size o_proj 把 [N, local_q_heads, head_dim] reshape 成 [N, Hq]，再经过 RowParallelLinear 变回 [N, H]。
Decode 路径不允许“没有 cache 也凑合跑”。当前实现要求 decode attention 一定绑定 layer KV cache，并且 Context 里必须给出 context_lens 和 block_tables，否则这条路径在语义上就是不完整的。

7. MLP、最终 Norm 与 LM Head gather last token

Attention 结束后，Qwen3MLP 把归一化后的 hidden 先扩成 gate 和 up 两路，再经过 SiLU 门控回到中间维度，最后下投回 hidden_size。模型尾部则执行最终 RMSNorm 和 ParallelLMHead。

Qwen3MLP 的三个步骤

gate_up_proj 用一次 GEMM 同时得到 gate 和 up 两路；SiluAndMul 计算 silu(gate) * up；down_proj 再把 [N, I] 投回 [N, H]，TP>1 时会做 all_reduce_sum。

Final Norm 的职责

最后一层返回的 hidden 还没有并入最终 residual，所以模型尾部会通过 norm_.forward(hidden_states, residual) 做最终一次融合和归一化，得到送往 lm head 的隐藏状态。

LM Head 为什么只看最后 token

在 prefill 且定义了 cu_seqlens_q 时，ParallelLMHead::forward 会先调用 gather_last_tokens，把整段 prompt 的 hidden 压成每条序列一个尾 token 表示，因此 logits 的行数是序列数，而不是 prompt token 总数。

Tensor gate_up = gate_up_proj_.forward(input, allocator, stream);
Tensor activated = act_fn_.forward(gate_up, allocator, stream);
Tensor mlp_out = down_proj_.forward(activated, allocator, stream);

if (residual.defined()) {
    hidden = norm_.forward(hidden_states, residual, allocator, stream).first;
} else {
    hidden = norm_.forward(hidden_states, allocator, stream);
}

Tensor logits = lm_head_.forward(hidden, allocator, stream);
LM Head 在 TP 下会先在每个 rank 上算本地 [rows, vocab_local] logits，再调用 communicator 的 gather_last_dim_to_rank0。这意味着 rank0 拿到完整 [rows, vocab_size]，而其他 rank 当前语义下拿到的是未定义张量。

8. Tensor Parallel 汇合点 collective

这份实现里的 TP 不是“只把参数切开，运行时还是单卡思维”。哪些层分片、哪些层汇总、哪些层只保留局部 shard，都是算子语义的一部分。

模块 本地持有的权重 本地 forward 输出 通信行为
VocabParallelEmbedding 词表按 vocab 维切 shard 本 rank 只对自己词表命中的 token 产生非零 embedding all_reduce_sum 合成完整 hidden。
QKVParallelLinear Q 头和 KV 头分别按输出维切 shard 只得到本 rank 的 q/k/v 头 无显式通信，后续 attention 和 o_proj 都在本地 shard 上计算。
RowParallelLinear 输入列按 input 维切 shard 每个 rank 算出完整输出形状的一部分和 all_reduce_sum 得到完整输出。
MergedColumnParallelLinear gate 和 up 各自按输出维切 shard 本 rank 拿到 gate_local 和 up_local 无显式通信，等到 down_proj 再聚合。
ParallelLMHead 词表按 vocab 维切 shard 本地 logits 只有 vocab_local 列 gather_last_dim_to_rank0，仅 rank0 拿完整 vocab 维。

communicator 的两种形态

当前代码同时支持 in-process communicator 和可选 NCCL backend。前者适合单进程测试里的 lockstep 执行，后者适合真正两线程、两设备的直接 forward。

为什么 group key 重要

model.embed_tokens、每层 self_attn.o_proj、每层 mlp.down_proj、lm_head 都带稳定的 TP group key。通信层依赖这个 key 把属于同一组 collective 的 rank 正确对齐。

9. 权重加载与算子映射

推理路径之所以能真正跑起来，不只是 forward 顺了，还因为 loader 能把 Hugging Face 风格的参数名和 packed 权重布局准确映射到这些算子实例上。

Packed 映射

q_proj、k_proj、v_proj 会映射到同一个 qkv_proj；gate_proj、up_proj 会映射到同一个 gate_up_proj。所以 forward 里看到的是一整个 qkv 张量和一整个 gate_up 张量，而不是多次独立 GEMM。

TP 下为什么要按 shard_id 加载

在 TP 场景里，qkv_proj 和 gate_up_proj 不能简单把一整块 contiguous 权重直接塞进去，必须像真实 loader 一样，分别按 q/k/v 或 gate/up 的 shard_id 把权重写入本 rank 对应的目标切片。

static const std::vector<PackedModuleMapping> mapping = {
    {"q_proj", "qkv_proj", 0},
    {"k_proj", "qkv_proj", 1},
    {"v_proj", "qkv_proj", 2},
    {"gate_proj", "gate_up_proj", 0},
    {"up_proj", "gate_up_proj", 1},
};
这也直接影响文档表达方式。既然代码里是“一次 GEMM，再切片解释”，那文档就不应该把 Q、K、V 或 gate、up 写成彼此独立的多次矩阵乘法，否则会和真实实现脱节。

10. 测试覆盖与当前限制

这份推理文档不是空口描述，当前仓库里的模型级测试已经用三层证据把它钉住了：单卡参考前向、TP 锁步回归、NCCL 直连前向。

测试主题 验证点 说明了什么
Qwen3ForCausalLmForwardAndLogitsMatchReference 极小模型下，从 embedding 到最终 logits 的数值结果与手工参考一致 证明单卡完整前向链路已经打通，不只是层级单测能过。
Qwen3TensorParallelModelMatchesSingleRankReference 两 rank TP 模型与单 rank 参考模型比对 embedding、hidden、logits 证明全模型 TP 语义已经串通，不只是层级通信函数单独正确。
Qwen3TensorParallelNcclDirectForwardMatchesSingleRankReference 两个线程、两个设备直接跑 Qwen3ForCausalLM::forward 并与单 rank 结果对齐 证明 communicator 不再需要测试代码手工 lockstep 才能完成端到端前向。
当前仍然需要记住的限制：
  • 很多 forward 路径都要求 CUDA、contiguous 和 float32。
  • Decode attention 必须先绑定 layer KV cache，并且 Context 里要有 context_lens 和 block_tables。
  • TP 下 lm_head 只有 rank0 会返回完整 logits，其他 rank 当前语义是 undefined tensor。
  • 如果 tp_size > 1 但没有安装 communicator，代码会直接抛异常，不再退回本地 shard 输出。

如果要用一句更工程化的话概括这条数据流，可以这样说：Qwen3ForCausalLM 把 embedding、预归一化残差流、页式 attention、门控 MLP 和 TP collective 串成了一条真实可运行的最小推理栈；模型层负责算子组合，Context 负责把序列边界、页表、KV 写入位置和 prefill / decode 语义注入运行时。
