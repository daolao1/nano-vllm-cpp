Config — 引擎运行时配置

控制推理引擎的资源分配、批处理策略和硬件配置。通过 Config::from_model_dir() 工厂方法创建。

model std::string 必需
HuggingFace 模型目录的本地路径。目录下必须包含 config.json 和 *.safetensors 权重文件。
示例：/home/user/huggingface/Qwen3-0.6B
max_model_len int 默认 4096
引擎允许的最大序列长度（prompt + 生成 token 总数）。 会被自动截断到模型支持的上限（hf_config.max_position_embeddings）。
影响：值越大，单个请求可用上下文越长，但 KV Cache 显存占用越多、可分配的 block 数越少。
初始化时自动执行 max_model_len = min(max_model_len, hf_config.max_position_embeddings)
max_num_seqs int 默认 512
每次 step() 调度的最大序列数（即最大并发 batch size）。 Scheduler 在 prefill 和 decode 阶段都会遵守此限制。
影响：值越大，吞吐量越高，但单次 step 的计算量和显存占用增大。
max_num_batched_tokens int 默认 16384
每次 step() 处理的最大 token 总数。仅在 prefill 阶段起作用 — Scheduler 不断从 waiting 队列取序列， 直到累计 token 数超过此值。
影响：控制 prefill 阶段的显存峰值。值越大，一次处理的 prompt 越多，但 GEMM 计算量线性增长。
初始化时自动确保 max_num_batched_tokens ≥ max_model_len（至少能处理一个最长序列）
gpu_memory_utilization float 默认 0.9
GPU 显存利用率上限（0, 1]。ModelRunner 在 warmup 后根据此值计算 KV Cache 可用空间：
kv_bytes = total_vram × gpu_memory_utilization - peak_usage
影响：值越高，KV Cache block 越多，可并发处理的序列越多；但留给模型计算的临时显存越少，有 OOM 风险。
tensor_parallel_size int 默认 1
张量并行度（TP），即使用的 GPU 数量。每张卡负责模型权重的 1/tp_size。
取值范围：[1, 8]。需要 NCCL 通信。
影响：TP=2 时每卡的 attention heads、MLP intermediate 维度减半；降低单卡显存但引入 all_reduce 通信开销。
enforce_eager bool 默认 false
设为 true 时禁用 CUDA Graph，所有 decode 步骤都用 eager 模式执行。
影响：CUDA Graph 在 decode 阶段通过录制和重放避免 CPU 端 kernel launch 开销，通常可提速 10-30%。 禁用后调试更方便（CUDA Graph 对动态逻辑有限制）。
kvcache_block_size int 默认 256
PagedAttention 的物理页大小（每个 block 存储的 token 数）。 每个 block 的显存占用 = 2(K+V) × num_layers × block_size × kv_heads × head_dim × dtype_bytes。
影响：值越大，prefix cache 命中的粒度越粗（需要完全匹配整个 block 的 token 才能复用）； 值越小，内存碎片越多、block 管理开销越大。
必须是 256 的倍数
num_kvcache_blocks int 默认 -1 自动计算
KV Cache 的总 block 数量。由 ModelRunner 在 allocate_kv_cache() 中根据 剩余显存自动计算，用户无需设置。
计算公式：(total × gpu_util - peak_used) / block_bytes
eos_token_id int 默认 -1 自动赋值
End-of-Sequence token 的 ID。由 from_model_dir() 从 hf_config.eos_token_id 自动赋值。 Scheduler 在 postprocess() 中检查生成的 token 是否等于此值来判断序列结束。
Qwen3-0.6B: 151645

HFConfig — 模型架构配置

从 HuggingFace 模型目录下的 config.json 自动解析。描述 Transformer 模型的结构参数。

hidden_size int 必需
模型的隐藏层维度（embedding 维度）。所有 attention 和 MLP 层的输入/输出维度。
Qwen3-0.6B: 1024  |  Qwen3-1.7B: 2048  |  Qwen3-8B: 4096
num_attention_heads int 必需
Query 的注意力头数。决定了 Q 投影的输出维度 = num_attention_heads × head_dim。 TP 时按 num_heads / tp_size 切分到各卡。
Qwen3-0.6B: 16
num_key_value_heads int 必需
KV 的注意力头数（GQA — Grouped Query Attention）。 当 num_key_value_heads < num_attention_heads 时，多个 Q head 共享同一组 KV head， 减少 KV Cache 显存占用。
Qwen3-0.6B: 8（每 2 个 Q head 共享 1 组 KV）
KV Cache 大小正比于此值而非 num_attention_heads。
num_hidden_layers int 必需
Transformer decoder 层数。KV Cache 总量 = 2 × num_layers × num_blocks × block_size × kv_heads × head_dim。
Qwen3-0.6B: 28
intermediate_size int 必需
MLP 的中间层维度。Qwen3 使用 SwiGLU 结构：gate_up_proj 输出维度 = 2 × intermediate_size， 经过 SiluAndMul 后变回 intermediate_size，再由 down_proj 投影回 hidden_size。
Qwen3-0.6B: 3072（通常约 hidden_size × 3）
vocab_size int 必需
词表大小。Embedding 层和 LM Head 的第一维。TP 时按 vocab_size / tp_size 切分。
Qwen3: 151936
max_position_embeddings int 必需
模型训练时支持的最大位置编码长度。RoPE 的 cos/sin cache 预计算到此长度。 Config.max_model_len 不能超过此值。
Qwen3-0.6B: 40960
head_dim int 默认 hidden_size / num_attention_heads
每个注意力头的维度。决定了 RoPE 的 rotary_dim、KV Cache 每 token 每层每 head 的存储大小。
Qwen3-0.6B: 128（= 1024 / 16 × 2，显式指定）
rms_norm_eps float 默认 1e-6
RMSNorm 的 epsilon 值，防止除零：x * rsqrt(mean(x²) + eps)。
Qwen3: 1e-6
rope_theta float 默认 10000.0
RoPE（Rotary Position Embedding）的基底频率。 频率公式：inv_freq = 1 / (theta ^ (2i / head_dim))。 值越大，高频衰减越慢，理论上支持更长的上下文。
Qwen3-0.6B: 1000000（100万，远大于原始 Transformer 的 10000）
tie_word_embeddings bool 默认 false
是否共享 embedding 和 LM Head 权重。为 true 时， lm_head.weight 直接指向 embed_tokens.weight，减少参数量。
Qwen3-0.6B: true（小模型常用此策略）
attention_bias bool 默认 false
QKV 投影的 Linear 层是否包含 bias。 为 false 时（Qwen3），attention 启用 QK-Norm（对 Q、K 各施加 per-head RMSNorm）。
QK-Norm 有助于稳定训练，是 Qwen3 相对于 Qwen2 的关键结构变化。
hidden_act std::string 默认 "silu"
MLP 的激活函数名称。Qwen3 使用 SwiGLU，即 SiluAndMul：
output = silu(gate) × up，其中 silu(x) = x × σ(x)。
torch_dtype std::string 默认 "bfloat16"
模型权重和推理计算的数据类型。决定了 KV Cache 的 dtype 和 GEMM 精度。
"bfloat16"：范围大（与 FP32 相同指数位），精度略低于 FP16，是大模型推理的主流选择。
KV Cache 每 token 每层每 head 占用 = head_dim × 2 bytes（bf16）。
eos_token_id int 默认 -1
End-of-Sequence token ID。生成此 token 时序列标记为结束（除非 ignore_eos=true）。
Qwen3: 151645（对应 <|endoftext|>）
bos_token_id int 默认 -1
Begin-of-Sequence token ID。tokenizer 编码时可能会在序列开头插入此 token。
Qwen3: 151643

SamplingParams — 采样参数

每个推理请求携带的采样配置。定义在 include/core/sampling_params.h。

temperature float 默认 1.0
采样温度。Sampler 将 logits 除以此值后再做 softmax：
probs = softmax(logits / temperature)
• temperature → 0：趋近贪心（确定性），总选概率最高的 token
• temperature = 1.0：标准采样，忠实于模型分布
• temperature > 1.0：更随机，分布更平坦
max_tokens int 默认 64
该请求最多生成的 token 数（不含 prompt）。达到此数量时序列强制结束。
num_completion_tokens = num_tokens - num_prompt_tokens
ignore_eos bool 默认 false
是否忽略 EOS token。设为 true 时，即使生成了 eos_token_id 也不终止， 继续生成直到 max_tokens。常用于 benchmark 场景测试吞吐量。

参数校验规则

Config::from_model_dir() 在初始化时执行以下校验：

1. model 目录必须存在且包含 config.json
2. kvcache_block_size % 256 == 0
3. 1 ≤ tensor_parallel_size ≤ 8
4. 0 < gpu_memory_utilization ≤ 1.0
5. max_model_len = min(max_model_len, hf_config.max_position_embeddings)
6. max_num_batched_tokens = max(max_num_batched_tokens, max_model_len)
