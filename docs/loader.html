1. 模块总览

一句话总结：这一层不是“读文件的小工具”，而是一条把磁盘权重安全搬运到运行时参数张量上的加载通道。它负责文件格式校验、名字改写、参数路由、shard 选择和必要的 BF16 到 float32 转换，但不负责真实 Qwen3 模型树的注册与装配。

文件解析 mmap

SafeTensorsFile::open 把 .safetensors 文件解析成一组 CPU Tensor 视图。正文数据通过 mmap 暴露，不会先复制到新的主机缓冲区。

名称分发 registry

ParameterRegistry 把权重名映射到具体 loader 回调。它不关心参数属于哪一层，只负责根据目标名调用对应的加载动作。

packed 改名 qkv / gate_up

HuggingFace 权重里拆开的 q_proj / k_proj / v_proj、gate_proj / up_proj 会先改成运行时的合并参数名，再附带 shard_id 交给目标层。

参数落盘

真正决定写入哪一段参数的是各层自己的 weight_loader 或 bias_loader。层对象负责 shape 校验、TP shard 切分和目标区间选择。

输入路径 │ ├── collect_safetensors_files(path) │ └── 得到稳定排序的 .safetensors 文件列表 │ ├── SafeTensorsFile::open(file) │ ├── 读 header_size │ ├── 解析 JSON header │ └── 为每个 entry 构造 CPU Tensor view │ ├── resolve_packed_parameter_name(source_name, mapping, shard_id) │ └── 源名字变成目标名字，可选地产生 shard_id │ ├── ParameterRegistry::load(...) │ └── 取到注册好的 loader 回调 │ └── layer.weight_loader(source, shard_id, allocator, stream) └── copy_tensor_to_parameter(...) -> 写入 CUDA 参数
解析层返回 CPU Tensor view 注册表层做名字到动作的路由 层对象决定 shard 与目标切片 复制层决定 dtype 是否可接受

2. 文件与类型地图

这一部分把当前 Step 6 相关的文件、类型和职责拆开。后续如果要接 Step 7 模型组装，基本也会从这张地图往下接。

文件 核心类型 / 函数 职责 当前定位

src/utils/loader.cpp SafeTensorsFile::open、ParameterRegistry::load、copy_tensor_to_parameter 实现文件解析、名字分发、复制和 BF16 转换 Step 6 的主逻辑

struct PackedModuleMapping {
    std::string source_name;
    std::string target_name;
    int shard_id = 0;
};

using ParameterLoadFn = std::function<void(const Tensor& source,
                                           std::optional<int> shard_id,
                                           DeviceAllocator& allocator,
                                           cudaStream_t stream)>;
  • PackedModuleMapping 只描述源段名、目标段名和 shard_id，不做切片。
  • SafeTensorsFile 负责“把文件变成可读取的 Tensor 视图”。
  • ParameterRegistry 负责“把目标参数名变成一次具体回调调用”。
  • copy_tensor_to_parameter 是最后的闸门，决定什么类型组合允许真的写进参数。

3. SafeTensors 解析流程 SafeTensorsFile::open

解析器的目标不是把整个文件复制出来，而是构造出一组由 mmap 支持的 CPU Tensor 视图。这样后面的层级 loader 可以直接拿这些视图做 shape 校验、切 shard 和复制。

SafeTensorsFile SafeTensorsFile::open(const std::string& path);

读头部

先读取文件前 8 字节的小端 header_size，并确认 header 没有越过文件末尾。

解析 JSON

把 header 文本交给 nlohmann::json，每个非 __metadata__ 条目都视为一个 tensor 规格。

托管 payload

用 Memory::make_owned 接管 payload 所在映射区域，让文件映射的生命周期由 Memory 控制，而不是依赖文件描述符存活。

构造 Entry

根据 dtype、shape 和 data_offsets 为每个条目生成 Entry.tensor，最终放入 entries_ 索引表。

open(path) ├── open + fstat ├── mmap entire file ├── header_size = read_le_u64(mapping[0:8]) ├── header_json = parse(mapping[8 : 8 + header_size]) ├── payload = mapping[8 + header_size : end] ├── data_memory = Memory::make_owned(payload, deleter = munmap(base)) └── for each tensor spec: validate dtype / shape / offsets tensor = Tensor::from_memory(data_memory, shape, dtype, element_offset) entries_[name] = Entry{..., tensor}
检查项 为什么要检查 失败时通常意味着什么
文件长度至少 8 字节 否则连 header_size 都读不出来 不是合法 safetensors，或文件损坏
header_size 不越界 避免越界解析 JSON 头部字段与真实文件长度不一致
条目必须有 dtype / shape / data_offsets 这是构造 Tensor 视图的最小信息集 header 结构不完整
data_offsets = [begin, end] 且 end >= begin 保证切片合法 header 描述错误
shape × dtype_size == data_size 确保 shape 与 payload 真正对应 权重条目大小不匹配
偏移按元素字节对齐 保证 Tensor 视图起点正确 格式生成阶段存在错位 
supported dtypes in parser
  F16  -> ScalarType::kFloat16
  BF16 -> ScalarType::kBFloat16
  F32  -> ScalarType::kFloat32
  I32  -> ScalarType::kInt32
  I64  -> ScalarType::kInt64
这里的“支持 dtype”只表示解析器能识别这些类型，不表示后面的参数复制层一定支持它们写入目标参数。是否能真正加载成功，最终还取决于 copy_tensor_to_parameter 的转换规则。

4. 注册表与 packed 映射 ParameterRegistry::load

ParameterRegistry 的设计重点是把“名字解析”和“层级写入”解耦。它知道如何从目标参数名找到一个回调，但不知道 QKV 内部怎么拼、GateUp 怎么排，也不知道 embedding 是否要和 lm_head 共用逻辑。

void ParameterRegistry::load(const std::string& path,
                             const std::vector<PackedModuleMapping>& packed_modules_mapping,
                             DeviceAllocator& allocator,
                             cudaStream_t stream = nullptr) const;
阶段 输入 输出 说明
收集文件 目录或单文件路径 稳定排序的 safetensors 文件列表 目录模式下会扫描顶层所有 .safetensors
打开文件 单个文件路径 SafeTensorsFile 之后按 key 顺序迭代条目
packed 改名 源参数名 目标参数名 + 可选 shard_id 普通权重会原名直通
注册表查找 目标参数名 回调函数 找不到会直接抛异常
调用 loader source Tensor、optional shard_id 参数写入完成 真正的形状和 shard 决策在层对象里做 
registry.register_parameter(
  "model.layers.0.self_attn.qkv_proj.weight",
  [&](const Tensor& source,
      std::optional<int> shard_id,
      DeviceAllocator& allocator,
      cudaStream_t stream) {
    qkv.weight_loader(source, *shard_id, allocator, stream);
  });

4.1 packed_modules_mapping 到底做了什么

HF 源模块段 运行时目标模块段 shard_id 最终含义
q_proj qkv_proj 0 写入 QKV 合并张量中的 Q 区间
k_proj qkv_proj 1 写入 K 区间
v_proj qkv_proj 2 写入 V 区间
gate_proj gate_up_proj 0 写入 MLP 合并张量中的 gate 区间
up_proj gate_up_proj 1 写入 up 区间
source_name = model.layers.0.self_attn.k_proj.weight │ ├── matches mapping "k_proj" -> "qkv_proj", shard_id = 1 ▼ target_name = model.layers.0.self_attn.qkv_proj.weight │ ├── registry.find(target_name) ▼ loader callback for qkv_proj.weight │ └── qkv.weight_loader(source_tensor, 1, allocator, stream)
名称匹配不是简单的任意子串替换，而是要求 source_name 落在参数段边界上。这样可以避免把更长名字里的偶然子串误替换掉。

5. 各层 weight_loader 行为 layer-owned logic

ParameterRegistry 只负责找到目标层，真正决定 source 是否可接受、如何切 shard、应该写到目标参数哪一段的是各层自己的 loader。这里表里的 output_size、local_output_size、input_size、local_input_size 都是层在构造时已经知道的目标尺寸；loader 只是拿 source.sizes() 去和这些已知尺寸做匹配，不是从 source 里“读出语义名”。

模块 可接受的 source 形状 内部动作 当前状态
ColumnParallelLinear weight: 第 1 维必须等于 input_size；第 0 维若等于 output_size 视为完整表，若等于 local_output_size 视为本地 shard
bias: 第 0 维若等于 output_size 视为完整表，若等于 local_output_size 视为本地 shard		 若 source.sizes()[0] == output_size，则沿第 0 维切出本 rank 对应区间；若 source.sizes()[0] == local_output_size，则直接复制 支持 weight / bias
RowParallelLinear weight: 第 0 维必须等于 output_size；第 1 维若等于 input_size 视为完整表，若等于 local_input_size 视为本地 shard
bias: 必须等于 [output_size]		 若 source.sizes()[1] == input_size，则沿第 1 维切输入 shard；若 source.sizes()[1] == local_input_size，则直接复制；bias 不切，直接复制 支持 TP 本地 shard 加载；前向阶段返回本 rank partial output
QKVParallelLinear 直接模式接受已打包形状；packed 模式下 Q 接受全量或 local Q shard，K / V 接受全量或 local KV shard 先选 source shard，再把它写到目标权重的 Q / K / V 连续区间 支持 weight / bias 两种 packed 重载
MergedColumnParallelLinear 直接模式接受总输出维；packed 模式下每个 shard 接受全量或 local shard 根据 shard_offsets_ 写入 gate / up 对应区间 支持 weight / bias 两种 packed 重载
VocabParallelEmbedding [vocab_size, hidden] 或 [local_vocab_size, hidden] 若 source 是全 vocab 权重，则按 rank 切 vocab 维 支持 TP 本地 vocab shard 加载；前向阶段返回本 rank 的 embedding 贡献
ParallelLMHead [vocab_size, hidden] 或 [local_vocab_size, hidden] 逻辑与 embedding 相同，按 vocab 维切本地 shard 支持 TP 本地 vocab shard 加载；前向阶段返回本 rank 的 local logits
换句话说，loader 不是“看 source 自己说我是完整表还是本地表”，而是“把 source.shape 和层构造时保存的 full/local 目标尺寸做比较”。例如 ColumnParallelLinear 判断完整表还是本地 shard，只看第 0 维是等于 output_size_ 还是 local_output_size_。

5.1 QKV 合并布局

destination weight shape:
  [local_q_size + 2 * local_kv_size, hidden_size]

shard_id = 0  -> rows [0, local_q_size)
shard_id = 1  -> rows [local_q_size, local_q_size + local_kv_size)
shard_id = 2  -> rows [local_q_size + local_kv_size,
                       local_q_size + 2 * local_kv_size)

5.2 GateUp 合并布局

output_sizes_       = [gate_size, up_size, ...]
local_output_sizes_ = [gate_local, up_local, ...]
shard_offsets_      = prefix sum(local_output_sizes_)

shard_id = i -> rows [shard_offsets_[i],
                      shard_offsets_[i] + local_output_sizes_[i])

ColumnParallelLinear

这类 loader 最简单。source 不是全量就是 local shard，判断依据只有第 0 维大小，最后调用 copy_tensor_to_parameter 即可。

QKVParallelLinear

它的关键点不是名字映射，而是 destination 的三段布局固定。q / k / v 的权重最终都会落进同一个目标 Tensor，只是写入的行区间不同。

MergedColumnParallelLinear

和 QKV 类似，但它不是固定三段，而是由 output_sizes_ 和 shard_offsets_ 决定每段区间，因此更适合泛化的 gate / up 合并场景。

Embedding / LM Head

这两类都是 vocab 维切分问题。接口本来就是按本地 vocab shard 设计的；现在加载与前向都能直接工作在 TP 本地 shard 语义上，只是最终的 all-reduce / gather 仍属于更上层的通信阶段。

6. 类型转换与复制规则 copy_tensor_to_parameter

解析器只负责把文件里的字节描述成 Tensor，真正决定“这份 Tensor 能不能进参数”的，是 copy_tensor_to_parameter。它先检查 defined、destination contiguous、shape 一致，再判断 dtype 组合是否被允许；如果 source 是 CPU 上的非连续 view，会先 materialize 成连续 host buffer，再执行复制或类型转换。

void copy_tensor_to_parameter(const Tensor& source,
                              const Tensor& destination,
                              DeviceAllocator& allocator,
                              cudaStream_t stream = nullptr);
源张量 目标张量 当前行为 为什么
CPU + 同 dtype CPU 连续 source 直接 memcpy；非连续 source 先 materialize 再复制 允许 loader 直接消费 slice/view 形式的 CPU shard
CPU + 同 dtype CUDA 连续 source 直接设备复制；非连续 source 先 materialize 成连续 host buffer，再走 DeviceAllocator::copy_to_device_async 像 RowParallelLinear 这种按输入维切 shard 的场景，source 常常是非连续 view
CPU BF16 CPU float32 先转成临时 std::vector<float>，再复制 这是当前 Step 6 的核心转换路径
CPU BF16 CUDA float32 先转成临时 float buffer，再拷到设备端 与现有 float32 参数实现对齐
其他组合 任意 直接抛异常 当前没有对应转换逻辑和测试保证 
if source.dtype() == destination.dtype():
    copy_same_dtype_tensor(...)
else if source.dtype() == ScalarType::kBFloat16 &&
        destination.dtype() == ScalarType::kFloat32:
    converted = convert_bfloat16_to_float32(...)
    copy converted to destination
else:
    throw unsupported conversion

6.1 BF16 到 float32 的位级扩展

uint32_t bits = static_cast<uint32_t>(bf16_value) << 16;
float out = 0.0f;
std::memcpy(&out, &bits, sizeof(out));

这里的含义是：BF16 保留了 float32 的高 16 位，把它左移回 float32 的高位后，低 16 位尾数补零，就得到了与原 BF16 数值等价的 float32 表示。这是目前实现中最需要、也最稳定的一条转换路径。

当前复制实现要求 source 和 destination 都是 contiguous。如果某个上层切片返回了非连续 view，这里会直接拒绝，而不是偷偷做一次隐式整理。
当前实现不支持 GPU 源张量，也不支持 float16 到 float32、int 到 float、或任意 dtype 间的通用转换。换句话说，Step 6 解决的是“现有仓库需要的最小可用路径”，不是通用 Tensor 转换器。

8. 错误路径与调试顺序

这套加载器当前选择的是严格失败策略，不会静默跳过未知参数或可疑 shape。所以文档必须把常见异常来源写清楚，不然排查时会很慢。

阶段 常见触发条件 代表了什么问题 建议先检查什么
collect_safetensors_files 路径不是目录也不是 .safetensors 文件 输入路径本身就不满足加载前提 模型目录、后缀名、目录层级
SafeTensorsFile::open header 越界、JSON 非对象、offset 越界、shape 不匹配 文件格式不合法或文件损坏 源 safetensors 是否完整、是否真由标准工具导出
resolve_packed_parameter_name 没有命中任何 packed 映射 这不是异常，说明条目按原名继续 确认该参数是否本来就不需要 packed
ParameterRegistry::load 目标参数名未注册 真实模型参数名与权重名字没有对齐 模块前缀、层号、packed 重命名结果
layer weight_loader rank / shape / shard_id 不合法 条目名字对上了，但布局规则没对上 full 尺寸、local 尺寸、tp_size、rank
copy_tensor_to_parameter 未定义、非 contiguous、shape 不一致、dtype 不支持 最终写入条件不满足 source 是否仍是连续 view，目标参数是否已初始化
调试顺序建议：先看失败发生在“名字找不到”还是“shape 不合法”，最后才看“复制失败”。早期实现里大多数问题并不在 CUDA 复制，而在参数命名和 shard 维度理解不一致。
当前实现不会“尽量加载能加载的部分”。只要某个条目找不到注册项，或者某个 loader 判断 shape 不对，整个 load 调用就会立刻失败。这是有意设计，因为在模型尚未装配完成的阶段，快速暴露第一处偏差比静默吞错更有价值。

9. 测试覆盖与讲解角度

目前的测试不是大而全的集成验证，而是把 Step 6 需要保证的几个关键点拆开做最小闭环。这样每个失败都更容易定位。

SafeTensorsParserExposesCpuTensorViews

验证 safetensors 文件可以被解析成正确的 CPU Tensor 视图，包括 keys、dtype、shape 和底层值。

BaseLinearWeightLoadersCopyFromCpuTensors

验证 ColumnParallelLinear 和 RowParallelLinear 可以接受 CPU float32 source，并把值准确写进设备侧参数。

ParameterRegistryLoadsPackedBfloat16Weights

验证 packed 改名、optional shard_id、QKV / GateUp 合并写入、embedding / lm_head 加载以及 BF16 到 float32 转换能联动成立。

测试名 证明了什么 没有证明什么

10. 当前限制与下一步

把边界说清楚，比在文档里假装“什么都支持”更有价值。下面这些不是偶然缺口，而是当前 Step 6 明确尚未覆盖的范围。

  • ParameterRegistry 目前还是通用分发层，真实 Qwen3 参数树尚未在模型类中注册。
  • copy_tensor_to_parameter 目前只支持 CPU source，转换路径只覆盖 BF16 到 float32。
  • RowParallelLinear、VocabParallelEmbedding、ParallelLMHead 已支持 TP 本地 shard 语义；真正的多卡 all-reduce / gather 仍属于后续模型装配与通信层工作。
  • 加载阶段对每次设备复制都会同步流，行为正确，但还不是最终高性能形态。
  • 缺少“加载真实 Qwen3-0.6B 并打印每层 sum / mean 与 Python 对比”的集成验证。
Step 6 已完成的边界 SafeTensors file -> CPU Tensor views -> packed name remap -> registry dispatch -> layer-owned shard selection -> BF16 -> float32 copy -> CUDA parameters Step 7 要承接的边界 real Qwen3 module tree -> register every parameter name -> connect packed mapping to actual model hierarchy -> run real forward and compare logits
最合理的下一步不是继续把 loader 层做得更花，而是直接在 Step 7 的模型组装里复用这套接口，把真实参数名注册到 Qwen3Attention、Qwen3MLP、Qwen3Model、Qwen3ForCausalLM 上。
