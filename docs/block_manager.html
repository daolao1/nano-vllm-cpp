1. 模块总览

一句话总结：BlockManager 不是“保存 token 的普通容器”，而是一个“页式 KV Cache 资源管理器”。Sequence 决定逻辑 token 序列，BlockManager 决定这些 token 的 KV 实际落在哪些物理页上，以及哪些页可以跨请求复用。

页式分配

把 KV Cache 切成固定大小的 block。每个 block 对应一页物理存储，避免整段序列连续扩容带来的拷贝成本。

前缀复用

通过 hash chain 判断完整 block 是否和已有请求共享前缀。如果命中，直接复用已存在的 KV，而不是重新 prefill。

引用计数

多个 Sequence 可以共享同一物理 block。Block 的 ref_count 控制真正的回收时机，避免过早释放共享页。

Decode 维护

decode 每轮通常只追加 1 个 token。BlockManager 负责判断是否要新开 block，以及一个残缺 block 何时升级为可复用缓存。

Sequence.token_ids [t0 t1 t2 ... t599] │ ├── 逻辑按 block_size 切分 ▼ 逻辑 block [0..255] [256..511] [512..599] │ ├── BlockManager.allocate() 生成映射 ▼ Sequence.block_table [7, 12, 3] │ │ │ │ │ │ │ └── 物理 block 3: 残缺 block, 暂不可复用 │ │ └────── 物理 block 12: 命中 prefix cache │ └────────── 物理 block 7: 命中 prefix cache ▼ 物理 KV pages blocks_[7], blocks_[12], blocks_[3]
Sequence 持有完整 token_ids BlockManager 持有物理页状态 Scheduler 通过它做资源判断 Attention kernel 通过 block_table 找 KV

2. 核心数据结构

理解这个模块最重要的是区分三个层次：逻辑 token 序列、逻辑 block 切分、物理 block 分配。

2.1 Block

struct Block {
    int block_id = 0;
    int ref_count = 0;
    uint64_t hash = kNoHash;
    std::vector<int32_t> token_ids;
};
字段 含义 为什么重要
block_id 物理 block 编号 Sequence.block_table 里记录的就是它，供 attention 和调度层引用。
ref_count 被多少条序列共享 prefix cache 命中后多个 Sequence 会指向同一个物理 block，只有引用归零才允许真正回收。
hash 当前 block 的 hash chain 值 是 prefix cache 的索引 key。完整 block 才有稳定 hash。
token_ids 该 block 对应的 token 内容 用于二次校验，避免 hash collision 导致错误复用。

2.2 BlockManager 内部状态

int block_size_;
std::vector<Block> blocks_;
std::unordered_map<uint64_t, int> hash_to_block_id_;
std::deque<int> free_block_ids_;
std::unordered_set<int> used_block_ids_;
  • blocks_ 是物理 block 池，大小在初始化时固定。
  • free_block_ids_ 记录当前空闲页，可被新序列或 decode 扩容使用。
  • used_block_ids_ 记录正在使用的页，和 free 队列互补。
  • hash_to_block_id_ 是 prefix cache 索引，key 是 hash chain，value 是物理 block id。

2.3 与 Sequence 的配合

std::vector<int32_t> token_ids;
int32_t num_cached_tokens = 0;
std::vector<int32_t> block_table;

这三个字段一起才能把“逻辑序列”和“物理页状态”接起来：

  • token_ids 负责表示完整上下文内容，BlockManager 用它切 block、算 hash、注册缓存。
  • num_cached_tokens 表示 prefix cache 已经覆盖的前缀长度，后续 prefill 只需要算 suffix。
  • block_table 是逻辑 block 到物理 block 的映射，attention kernel 需要靠它定位 KV 页。

3. 哈希链与 Prefix Cache

BlockManager 不是直接拿整个 prompt 做一次 hash，而是按 block 构造 hash chain。这样可以做到“前缀命中到第几个完整 block”为止，而不要求整段 prompt 完全相同。

uint64_t compute_hash(const std::vector<int32_t>& token_ids,
                      uint64_t prefix = kNoHash);
h0 = kNoHash h1 = hash(block0, h0) h2 = hash(block1, h1) h3 = hash(block2, h2) 如果两条序列拥有相同的前两个完整 block: seqA: block0 == seqB.block0 seqA: block1 == seqB.block1 那么 h1 和 h2 必然相同, 前两个 block 可复用。 但一旦 block2 不同: h3 不同, 从这一块开始之后的 block 都不能复用。

3.1 为什么只缓存完整 block

  • 残缺 block 的内容还会继续增长，hash 不稳定。
  • 如果对残缺 block 提前注册缓存，每次 decode append 都要更新 hash 索引，管理复杂度明显上升。
  • 按 block 粒度缓存会损失最多一个 block 的命中精度，但能换来简单、稳定的实现。
当前实现中，最后一个不完整 block 会把 hash 设为 kNoHash。这意味着“只命中完整块，不命中残缺块”是明确设计，而不是偶然行为。

3.2 为什么还要比较 token_ids

查到 hash 命中后，代码还会进一步比较 blocks_[block_id].token_ids != token_ids。这是为了避免极低概率的 hash collision 造成错误复用。也就是说，hash 只负责快速定位，token 内容校验才是最终确认。

实现里采用的是“一旦 miss，后续全 miss”的策略。原因是当前请求的 block_table 必须连续成立，不能跳过中间一段之后再复用后面的块，否则逻辑前缀就断裂了。

4. Prefill 路径 allocate / can_allocate

Prefill 阶段的核心任务是：为一条新 Sequence 建立 block_table，同时尽可能复用已经存在的 KV Cache 前缀。

4.1 can_allocate 在判断什么

bool BlockManager::can_allocate(const Sequence& seq) const {
    return free_block_ids_.size() >= seq.num_blocks();
}

这个实现是一个偏保守的资源检查：它只按“整条序列需要多少 block”估算，不提前把 prefix cache 命中节省的 block 算进去。优点是简单、不会高估可分配能力；缺点是理论上会错过一小部分本可以依赖 cache 命中的请求。

4.2 allocate 的执行过程

for each logical block in Sequence: 1. 取出该 block 的 token_ids 2. 如果是完整 block, 用 prefix hash 计算当前 hash 3. 去 hash_to_block_id_ 查缓存 4. 若命中且 token 内容完全一致: - seq.num_cached_tokens += block_size - block.ref_count++ 或重新激活 block 否则: - 从 free_block_ids_ 拿一个新物理 block 5. 把 block_id 写入 seq.block_table 6. 如果当前 block 是完整 block, 更新 hash_to_block_id_

4.3 一个具体例子

假设 block_size = 256，序列长度为 612：

逻辑序列: block0 = tokens[0..255] 完整 block1 = tokens[256..511] 完整 block2 = tokens[512..611] 残缺 先到的 seq1: allocate -> 分配物理 [4, 9, 2] num_cached_tokens = 0 后到的 seq2, 前 512 token 和 seq1 完全相同: block0 命中 -> 复用 4 block1 命中 -> 复用 9 block2 不命中 -> 新分配 7 结果 block_table = [4, 9, 7] num_cached_tokens = 512

这时 prefill 的真实工作量不是 612，而是 612 - 512 = 100。调度器和输入准备都应该把前 512 个 token 当成已经完成的历史工作。

4.4 对输入准备的直接影响

  • prefill 的 query 只处理未命中的 suffix，例如 token_ids[512..611]。
  • position ids 仍要从 512 开始，而不是从 0 重新编号。
  • attention 的 key/value 长度仍然是完整上下文长度，因为模型需要看到完整历史。
  • slot mapping 只为新写入的 token 生成，因为命中的前缀 block 已经有现成 KV。

5. Decode 路径 can_append / may_append

Decode 阶段不再处理整段 prompt，而是围绕“新追加的 1 个 token 会不会跨 block 边界”展开。BlockManager 在这个阶段更像一个轻量的页维护器。

5.1 can_append 的判断逻辑

int need = (seq.len() % block_size_ == 1) ? 1 : 0;

调用时机是在 append_token() 之后。此时如果长度从 256 -> 257，那么新 token 已经落到下一个 block，需要 1 个新物理页；否则还写在当前残缺 block 里，不需要额外页。

追加后长度 len % block_size 含义 是否要新 block
256 0 最后一个 block 刚好填满 否，但要注册 hash
257 1 新 token 溢出到下一个 block 是
258..511 2..255 仍写在当前残缺 block 否

5.2 may_append 的三种分支

if (rem == 1) {
    // 上一个 block 已满, 为新 token 分配新页
} else if (rem == 0) {
    // 当前 block 刚填满, 计算 hash 并注册进 cache
} else {
    // 当前 block 仍残缺, 暂不做任何事
}
情况 A: rem == 1 原来长度 = 256 append 1 个 token 后 = 257 新 token 落在新 block, 所以 block_table 追加一个物理页 情况 B: rem == 0 原来长度 = 511 append 1 个 token 后 = 512 最后一个残缺 block 被填满, 现在可以生成稳定 hash 并允许后续请求复用 情况 C: rem == 2..255 原来长度 = 300 append 1 个 token 后 = 301 只是往当前残缺 block 继续写, 既不扩容也不注册缓存

5.3 为什么 decode 阶段不能把当前长度等同于缓存长度

因为 decode 每轮都会增长 num_tokens，但增长部分通常落在最后一个残缺 block，直到填满之前都不能视为稳定可复用缓存。所以模块里才会同时保留：

  • num_tokens：当前真实长度，决定下一个 token 写在哪。
  • num_cached_tokens：已经命中或稳定可复用的前缀长度，决定 prefill 可跳过多少工作。
  • num_cached_blocks()：一个快速的整块视角，便于上层按 block 粒度思考命中范围。

6. 关键不变量与边界条件

这些不变量是面试回答和调试排错时最值得强调的点。理解它们，基本就抓住了 BlockManager 的设计核心。

不变量 含义 如果破坏会怎样
seq.block_table.size() == seq.num_blocks() 逻辑 block 和物理映射一一对应 attention 读取 KV 时会找不到页，或者页数不匹配。
num_cached_tokens 是 block_size 的整数倍 当前实现只缓存完整 block 如果不是整块，就意味着残缺 block 被错误地当成稳定缓存。
hash == kNoHash 表示不可复用 典型场景是残缺 block 或新分配未填满 block 错误注册会导致后续请求命中不稳定前缀。
释放依赖 ref_count 共享 block 必须等所有引用都结束 过早释放会把仍在使用的 KV 页放回空闲池，造成数据破坏。
一旦 miss, 后续全 miss prefix 复用必须是连续前缀 不能出现“中间一块不同, 后面又继续复用”的断裂映射。

常见边界条件

  • 单 token 序列仍然占 1 个 block。
  • 恰好填满一个 block 时，不会新分配 block，但会注册 hash。
  • 从完整 block 跨到新 block 的那一步，can_append() 才需要检查空闲页。
  • deallocate() 会把 seq.num_cached_tokens 清零，并清空 block_table，表明该 Sequence 已不再持有任何物理页。

7. 测试覆盖与讲解角度

当前仓库已经有一套比较完整的单元测试。阅读测试文件时，可以把它们当作“BlockManager 的行为规格说明”。

测试主题 验证点 你在讲解时可以怎么说
ComputeHashDeterministic 相同 token 输入得到相同 hash prefix cache 的索引必须稳定，否则无法跨请求复用。
PrefixCacheHit 共享前缀时前几个 block 直接复用 说明 num_cached_tokens 不是装饰字段，而是决定真实 prefill 工作量的元数据。
DeallocateRecyclesBlocks 共享 block 只有 ref_count 归零才回收 体现 block 生命周期管理，而不只是“把 id 放回 free list”。
MayAppendAllocatesNewBlock decode 跨 block 时要扩容 说明 decode 和 prefill 的关注点完全不同，decode 更关注尾部页是否溢出。
MayAppendRegistersHashWhenBlockFull 残缺 block 填满后才能进入 prefix cache 这是“只缓存完整块”这一设计最直接的行为证明。

面试式总结

如果要用一句更工程化的话概括这个模块，可以这样说：

BlockManager 是 Scheduler 和 attention kernel 之间的页式资源中间层。它把 Sequence 的完整 token 序列切成固定大小的逻辑块，再映射到可共享、可回收的物理 KV 页；prefill 阶段负责找可复用前缀并建立 block_table，decode 阶段负责判断尾部是否扩容以及何时把残缺块提升为可缓存块。
