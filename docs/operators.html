算子层文档

这一层位于 engine 与 Tensor/Memory Runtime 之间，负责把调度器给出的 batch 语义翻译成真正的张量计算。它既包含无状态的基础 kernel， 也包含带权重或带缓存状态的算子对象。本文档同时说明架构目标与当前仓库中的实际实现状态，避免把未来规划误认为现状。

当前阶段: 最小可用单卡实现 phase-1 kernel: 真实 CUDA kernel Attention: 设备端 CUDA 实现 Linear / Embedding / Sampler: 功能性 fallback Tensor runtime 术语: Memory

层次定位

在 nano-vllm-cpp 里，调度器只决定“这一轮要算哪些 token、要不要抢占、KV block 怎么映射”；真正的数值计算发生在算子层。 算子层的职责不是管理队列，而是吃掉已经排好版的 Tensor，然后产出下一层需要的 Tensor 或 logits。

上游 Scheduler / Sequence / BlockManager 负责 batch 组织、block 分配和运行时状态切换。
中间层 算子层消费 Context 与张量，封装 Attention、Linear、Embedding、Sampler 等模型计算单元。
下游 Tensor、Memory、CudaAllocator、cuBLAS wrapper 提供设备内存与基础计算载体。
engine/scheduler -> utils/context -> layers/* -> utils/tensor + utils/cuda_allocator + utils/cublas_wrapper

组件总览

当前公开接口可以分成两类: 无状态 kernel 函数，以及带参数或缓存的状态型算子。

模块 接口 职责 当前实现
Context get_context / set_context / reset_context 把 prefill/decode 的批处理元信息作为线程本地上下文传给 Attention、LM Head 等算子。 已实现
kernel_ops rms_norm, add_rms_norm, rotary_embedding, silu_and_mul, store_kvcache 最基础的逐 token / 逐 row CUDA kernel。 真实 CUDA kernel
Attention set_kv_cache, forward 管理 KV cache 引用，并根据 Context 走 prefill 或 decode 路径。 设备端 CUDA 实现，当前为朴素版 paged/dense attention
Linear 家族 Column / Row / QKV / MergedColumn 承载线性层权重，并提供 TP 语义上的切分边界。 功能可用，当前为 host-side fallback
Embedding / LM Head VocabParallelEmbedding, ParallelLMHead 词嵌入查表与输出 logits，LM Head 在 prefill 下只取每个序列最后一个 token。 功能可用，当前为 host-side fallback
Sampler forward 对 logits 做 temperature 调整并采样下一个 token。 功能可用，当前为 host-side fallback

Context 侧带信息

Attention 和 LM Head 不是只看输入张量 shape 就能判断当前批次语义。prefill 需要知道每个序列的边界，decode 需要知道每个请求已经积累了多少上下文。 因此仓库没有把这些元信息全部塞进 forward 参数，而是放在 thread_local Context 里。

字段 含义 主要消费者
is_prefill 标记当前 batch 是 prompt 阶段还是 decode 阶段。 Attention, ParallelLMHead
cu_seqlens_q / cu_seqlens_k 压缩 batch 的前缀和边界，告诉算子每个序列在拼接张量里的起止位置。 Attention, ParallelLMHead
max_seqlen_q / max_seqlen_k 最大 query / key 序列长度，主要是给高性能 attention 后端预留接口。 Attention 的未来优化版本
slot_mapping 每个 token 应写入哪一个 KV cache 槽位。 store_kvcache, Attention
context_lens decode 时每个请求当前可见的历史长度。 Attention
block_tables 逻辑 block 到物理 block 的映射表，用于 paged KV cache 回读。 Attention
set_context(...) -> thread_local g_context -> Attention::forward / ParallelLMHead::forward 读取

Phase-1 基础 Kernel

这些 kernel 是目前算子层里最接近生产实现的一部分。它们直接在 CUDA 上运行，不依赖 host-side 数值回退，主要覆盖 Transformer block 的公共算子。

rms_norm 对最后一维做 row-wise RMSNorm，输出 shape 与输入一致。
真实 CUDA
add_rms_norm 先做 residual add，再做 RMSNorm，并额外写出 residual_out 供后续层复用。
真实 CUDA
rotary_embedding 对 query / key 的 head_dim 维做 RoPE 旋转，支持按 position 查 cos/sin cache。
真实 CUDA
silu_and_mul 把 gate_up 按最后一维一分为二，执行 SiLU(gate) * up。
真实 CUDA
store_kvcache 按 slot_mapping 把每个 token 的 key/value 写到 paged cache 对应槽位。
真实 CUDA
phase1_kernel_specs 提供一个轻量说明表，便于测试、文档或初始化阶段枚举当前 kernel 能力。
元数据接口
实现要点
  • RMSNorm 与 AddRMSNorm 使用 warp reduction + block reduction，避免每一行都走全局同步。
  • float32 场景下为 RMSNorm / AddRMSNorm 提供了 float4 向量化 fast path。
  • RoPE 在 float32 场景下提供了 packed float2 fast path，减少标量读写与指令数。
  • kernel 接口统一要求输入在 CUDA 上且 contiguous，避免 view 语义把地址计算复杂化。
  • 支持的核心浮点 dtype 是 float16 / bfloat16 / float32；当前没有 INT8 量化路径。
典型调用关系
RMSNorm: hidden -> rms_norm(hidden, weight, eps, out) MLP gating: gate_up -> silu_and_mul(gate_up, out) Attention cache write: key/value + slot_mapping -> store_kvcache(...) RoPE: positions + cos_sin_cache + query/key -> rotary_embedding(...)

状态型算子

这些类承载权重、维度配置或缓存引用。接口形式已经和后续更高性能版本兼容，但当前实现优先保证端到端可运行，而不是吞吐最优。

Attention

Attention 持有 num_heads、num_kv_heads、head_dim、block_size 以及可选的 k_cache / v_cache。 forward 时会先做形状校验；如果 KV cache 与 slot_mapping 已设置，就先调用 store_kvcache 把本轮 token 写入缓存。 随后它直接在 CUDA 上按 prefill 或 decode 分支执行朴素 softmax attention，不再把大块 query/key/value 拷回 host。

  • prefill: 通过 cu_seqlens_q / cu_seqlens_k 切出每个序列，并在因果掩码范围内做注意力计算。
  • decode: 通过 context_lens + block_tables 从 paged KV cache 回读完整历史，再对单 token query 做注意力。
  • 当前只支持 contiguous CUDA float32 张量。
  • 已经不是 host-side fallback，但还不是 FlashAttention 的高性能版本。
当前为设备端 CUDA 实现
Linear 家族

ColumnParallelLinear、RowParallelLinear、QKVParallelLinear、MergedColumnParallelLinear 当前都复用一个 run_linear 辅助函数。 这个辅助函数会把 input / weight / bias 复制到 host，在 CPU 上完成矩阵乘加，再把结果拷回 CUDA Tensor。

  • ColumnParallelLinear 负责输出维切分，local_output_size = output_size / tp_size。
  • RowParallelLinear 已保留接口语义，但当前仅支持 tp_size = 1，还没有 all-reduce 汇聚路径。
  • QKVParallelLinear 本质上是一个特定输出布局的 ColumnParallelLinear。
  • MergedColumnParallelLinear 把多个输出分支拼接成一次线性映射。
  • 当前实现只支持 contiguous CUDA float32 张量，且未接入 cuBLAS GEMM。
当前为 host-side fallback
Embedding 与 LM Head

VocabParallelEmbedding 当前仍以单卡优先，要求 tp_size = 1。它会把 input_ids 拷到 host，执行查表后写回输出 Tensor。 ParallelLMHead 同样走 host-side matmul，但做了一件对推理链路很重要的事: 如果 Context 标记为 prefill 且给出了 cu_seqlens_q， 它只会抽取每个序列最后一个 token 的 hidden state 参与 logits 计算，这与真实推理时“prefill 只对最后位置采样”一致。

  • Embedding: 输出 shape = input_ids.shape + [hidden_dim]。
  • LM Head: 输入期望为 [N, hidden_dim]，输出为 [rows, local_vocab_size]。
  • 未来如果接入 tensor parallel，需要引入 vocab shard gather / reduce 逻辑。
当前为 host-side fallback
Sampler

Sampler::forward 接收 logits 和 temperatures，先在 host 上做 temperature 缩放与 softmax，再使用 Gumbel-max 风格的采样方式选出 token。 返回值不是 Tensor，而是 std::vector<int32_t>，因为采样结果通常马上回到调度器和序列状态机中，而不是继续参与 GPU 计算图。

  • logits 形状要求是 [batch_size, vocab_size]。
  • temperatures 形状要求是 [batch_size]。
  • 每一行使用 seed + row 构造确定性随机源，方便测试复现。
当前为 host-side fallback

Prefill / Decode 数据流

同一套算子接口，在 prefill 和 decode 下看到的语义并不一样，区别主要来自 Context 与 KV cache。

Prefill: input_ids -> Embedding -> QKV Linear -> RoPE -> Attention(write KV + causal attention) -> Output Linear -> MLP Linear -> SiLUAndMul -> Linear -> RMSNorm -> LM Head(select last token per sequence) -> Sampler
Decode: last_token_ids -> Embedding -> QKV Linear -> RoPE -> Attention(write current KV + read paged KV cache) -> Output Linear -> MLP -> RMSNorm -> LM Head -> Sampler
Prefill 关注点 需要用 cu_seqlens_q / cu_seqlens_k 表示一个压缩后的变长 batch；Attention 只允许每个 query 看到自己之前的 token；LM Head 只取每个序列最后一个位置。
Decode 关注点 query 往往只有当前步的一个 token，但 key/value 要从 paged KV cache 里按 block_tables 回读完整历史；context_lens 决定每个请求这一步到底能看到多长上下文。

当前实现状态

如果从“是否能跑通一轮单卡生成”这个标准看，算子层已经具备端到端能力；如果从“是否接近 vLLM 的高性能实现”这个标准看，还处在过渡阶段。

能力 架构目标 当前仓库状态
基础归一化 / 激活 / RoPE / KV 写入 直接走 CUDA kernel 已完成，且做了向量化优化
Attention FlashAttention C API + PagedAttention 路径 接口、KV 写入和数值计算都已在设备端打通，但实现仍是朴素 attention，不是 FlashAttention
Linear 家族 cuBLAS / cutlass + tensor parallel 通信 目前是 host-side matmul；RowParallel 仅支持 tp_size = 1
Embedding / LM Head 并行词表分片与设备端计算 功能可用，但当前仍是单卡 host fallback
Sampler 设备端采样或更高效的并行实现 当前在 host 上做 softmax + Gumbel-max
量化 dtype 支持 INT8 / FP8 等低精度路径 尚未实现；当前核心路径以 float16 / bfloat16 / float32 为主
这版实现已经解决的问题 Scheduler、Context、KV slot_mapping、block_tables、prefill/decode 语义、LM Head 最后 token 选择，这些推理框架层面的关键约束已经被编码进算子边界里。
下一阶段最值得替换的点 把 Attention 切到真正的 FlashAttention / PagedAttention 后端，把 Linear / LM Head 切到 cuBLAS 或 cutlass，把 TP 通信语义从接口落实到实现。
