1. 模块总览

一句话总结：这里的 Tensor 不是一个完整的通用框架，而是一个为推理引擎服务的最小张量运行时。它负责把“数据指针 + 形状 + 步长 + dtype + device”打包成统一抽象，并把真实内存分配委托给 allocator。

轻量元数据层

Tensor 自己不做复杂算子调度，主要承担形状、步长、dtype、device、memory_offset 这些元数据管理工作。

Memory 共享底座

多个 Tensor 视图可以共享同一块 Memory。reshape 和 slice 都是 view 语义，不主动拷贝底层数据。

CUDA 优先

当前运行时的动态分配路径主要围绕 CUDA 展开，真正的分配入口是 CudaAllocator，CPU 更像是测试和 borrowed memory 场景。

推理友好

它的设计目标是给 kernel、cuBLAS、KV Cache 和层实现提供稳定的底层接口，而不是覆盖自动求导、广播或高级索引。

DeviceAllocator.allocate(bytes, device) │ ▼ Memory │ + bytes + device + deleter │ ▼ Tensor(memory, sizes, strides, dtype, memory_offset) │ ├── reshape() 只改 sizes/strides ├── slice() 只改 sizes/memory_offset └── data() = base_ptr + memory_offset * element_size
Tensor 拥有的是元数据 Memory 决定底层内存生命周期 Allocator 决定在哪个设备上分配 view 共享同一底层指针

2. 类型系统与元数据

Tensor 模块先定义了最基本的设备和数据类型系统。后面所有 shape、memory、kernel 接口都建立在这几种枚举之上。

2.1 DeviceType / ScalarType

enum class DeviceType {
    kCPU,
    kCUDA,
};

enum class ScalarType {
    kFloat16,
    kBFloat16,
    kFloat32,
    kInt32,
    kInt64,
};
类型 当前支持 实现含义
DeviceType kCPU、kCUDA 只区分 CPU 与 CUDA，device index 放在 Device 结构里。
ScalarType fp16、bf16、fp32、int32、int64 既决定元素宽度，也被上层 kernel 和 GEMM 用来做输入合法性校验。

2.2 Device 结构

struct Device {
    DeviceType type = DeviceType::kCPU;
    int index = 0;
};

这里把 device 表达成 (type, index) 二元组。例如 GPU 0 就是 {DeviceType::kCUDA, 0}。operator== 和 operator!= 让它可以直接参与类型检查和错误判断。

2.3 大小与字符串映射

模块内部提供了 to_string(DeviceType)、to_string(ScalarType) 和 scalar_type_size(ScalarType)。其中最关键的是元素字节数映射：

float16 -> 2 bytes bfloat16 -> 2 bytes float32 -> 4 bytes int32 -> 4 bytes int64 -> 8 bytes Tensor.nbytes() = Tensor.numel() * Tensor.element_size()
如果后续要支持 fp8、bool、uint8 或者自定义量化类型，这一层是最先要扩展的地方，因为 stride、offset、nbytes 都依赖元素字节数。

3. Memory 的所有权模型 ownership

Memory 是这个 Tensor runtime 最关键的底座。Tensor 并不直接持有裸指针，而是通过 std::shared_ptr<Memory> 间接持有底层内存。

class Memory {
public:
  using Ptr = std::shared_ptr<Memory>;

    static Ptr make_owned(void* data,
                          size_t bytes,
                          Device device,
                          std::function<void(void*)> deleter);
    static Ptr make_borrowed(void* data, size_t bytes, Device device);
};

3.1 两种构造语义

接口 语义 典型用途
make_owned Memory 在析构时调用 deleter 回收底层内存 GPU tensor、allocator 返回的正式资源
make_borrowed Memory 只是包一层指针，不负责释放 测试里的 host array、零字节张量、外部缓冲区视图
owned memory: data ptr ----> allocated buffer │ └─ destructor calls deleter(ptr) borrowed memory: data ptr ----> external buffer │ └─ destructor does nothing

3.2 为什么 Memory 要和 Tensor 分开

  • view Tensor 可以共享一份底层内存，不需要重复分配。
  • slice() 只要改 offset，不需要复制 buffer。
  • 生命周期和元数据职责清晰分离，后续接 allocator、KV Cache、权重映射都更直接。
如果没有 Memory 这一层，Tensor 一旦支持 view，就会面临“谁负责 free 原始内存”的问题。多个 view 指向同一片数据时，用 shared ownership 明显比裸指针安全。

4. Tensor 的核心语义

Tensor 本体只保存五类核心字段：memory、sizes、strides、dtype、memory_offset。理解这五个字段，就理解了整个张量抽象。

class Tensor {
private:
  Memory::Ptr memory_;
    std::vector<int64_t> sizes_;
    std::vector<int64_t> strides_;
    ScalarType dtype_ = ScalarType::kFloat32;
  size_t memory_offset_ = 0;  // measured in elements
};

4.1 五个字段各管什么

字段 含义 为什么重要
memory_ 底层内存拥有者 决定 data pointer 和内存生命周期。
sizes_ 每个维度长度 决定 rank、numel 和 shape。
strides_ 每个维度前进 1 步时跨过多少元素 决定张量是否连续，也决定 view 如何映射到底层存储。
dtype_ 元素类型 影响 element_size、nbytes 和 kernel 解释方式。
memory_offset_ 相对 memory 起点的元素偏移 slice 不拷贝数据时就靠它表达“从哪开始看”。

4.2 data() 的真实含义

void* Tensor::data() const {
    auto* base = static_cast<std::byte*>(memory_->data());
    return base + memory_offset_ * element_size();
}

这里非常关键：memory_offset_ 的单位不是字节，而是元素个数。真正取地址时才乘上 element_size()。这样同一个 offset 逻辑可以统一服务于 fp16、fp32、int64 等不同 dtype。

4.3 三个构造入口

接口 行为 适用场景
Tensor::empty 向 allocator 申请未初始化内存 输出 buffer、中间张量
Tensor::zeros 先 empty，再 memset_async(..., 0) 需要确定初值为 0 的张量
Tensor::from_memory 复用已有 Memory，构造 view tensor host 测试数据、外部 buffer 包装
默认构造的 Tensor() 是 undefined tensor。它没有 memory，defined() 为 false，data() 返回 null，常用于可选参数和空上下文占位。

5. 连续布局与视图变换 layout view

这部分是 Tensor 模块最像“框架底层”的地方。虽然实现很小，但已经把 contiguous layout 和 view 语义分开了。

5.1 连续步长的生成方式

std::vector<int64_t> make_contiguous_strides(const std::vector<int64_t>& sizes)
例子: sizes = [2, 3, 4] strides[2] = 1 strides[1] = 1 * 4 = 4 strides[0] = 4 * 3 = 12 结果: [12, 4, 1]

这正是标准 row-major contiguous layout。测试里也明确验证了 {2,3,4} 对应的 stride 是 {12,4,1}。

5.2 numel 的定义

size_t compute_numel(const std::vector<int64_t>& sizes)
  • 所有 size 必须非负，否则抛出 std::invalid_argument。
  • 空 shape 返回 1，这意味着它支持标量语义。
  • 普通 shape 返回所有维度长度的乘积。

5.3 is_contiguous 的判断

实现方式非常直接：把当前 strides_ 和同 shape 下的标准 contiguous strides 做一次逐元素比较。两者完全相等，才视为连续。

5.4 reshape 为什么要求 contiguous

Tensor Tensor::reshape(const std::vector<int64_t>& sizes) const {
    if (!is_contiguous()) {
        throw std::invalid_argument("reshape requires a contiguous tensor");
    }
    if (compute_numel(sizes) != numel()) {
        throw std::invalid_argument("reshape must preserve numel");
    }
    return Tensor(memory_, sizes, make_contiguous_strides(sizes), dtype_, memory_offset_);
}

这里选择的是最保守、最清晰的规则：只有连续张量允许 reshape。这样不用引入更复杂的 stride 推导逻辑，也避免“看起来 reshape 成功、但底层索引已经不符合预期”的隐患。

5.5 slice 的语义

Tensor Tensor::slice(int64_t dim, int64_t start, int64_t end) const
原 tensor: sizes = [2, 3, 4] strides = [12, 4, 1] offset = 0 slice(dim=1, start=1, end=3) 后: sizes = [2, 2, 4] strides = [12, 4, 1] // 保持不变 offset = 0 + 1 * 4 = 4 // 单位仍然是元素 结果 view 从 memory 的第 4 个元素开始看

测试里正是用这个例子验证了：

  • sliced.memory() == tensor.memory()，说明底层共享。
  • sliced.memory_offset() == 4，说明 offset 是元素单位。
  • sliced.data_as<float>() == host.data() + 4，说明 data pointer 发生了偏移。
  • sliced.is_contiguous() == false，说明这个 view 不再是标准连续布局。
当前 slice 只支持单维区间切片，不支持 step、负索引、多维高级索引，也不会自动 materialize contiguous copy。上层 kernel 如果要求 contiguous，需要显式保证输入满足。

6. 与 CudaAllocator 的协作

Tensor 自己不关心具体如何 malloc，它只依赖 DeviceAllocator 接口。当前真正实现的是 CudaAllocator。

class DeviceAllocator {
public:
    virtual Memory::Ptr allocate(size_t bytes, Device device) = 0;
    virtual void memset_async(void* ptr, int value, size_t bytes, cudaStream_t stream) const = 0;
    virtual void copy_to_device_async(void* dst, const void* src, size_t bytes, cudaStream_t stream) const = 0;
    virtual void copy_to_host_async(void* dst, const void* src, size_t bytes, cudaStream_t stream) const = 0;
    virtual void synchronize_stream(cudaStream_t stream) const = 0;
};

6.1 Tensor::empty 到底做了什么

Tensor::empty(sizes, dtype, device, allocator) -> bytes = compute_numel(sizes) * scalar_type_size(dtype) -> memory = allocator.allocate(bytes, device) -> return Tensor(memory, sizes, contiguous_strides, dtype, 0)

6.2 zeros 的零填充路径

Tensor::zeros 在 empty 之后会调用 allocator.memset_async(tensor.data(), 0, tensor.nbytes(), stream)。也就是说，它的零初始化是否异步、在哪个 stream 上生效，完全交给 allocator 实现。

6.3 CudaAllocator 的当前约束

行为 实现 结果
分配 cudaMalloc 只支持 DeviceType::kCUDA，CPU 分配会直接抛异常。
释放 Memory deleter 里调用 cudaFree 和 Memory 生命周期自动绑定。
置零 cudaMemsetAsync 适合 Tensor::zeros 和中间 buffer 初始化。
Host/Device copy cudaMemcpyAsync 测试和 host fallback 都依赖这条链路。
同步 cudaStreamSynchronize 测试里会在 copy 后显式同步，确保结果可读。

6.4 零字节张量的处理

如果 bytes == 0，CudaAllocator 不会去调用 cudaMalloc，而是返回一个 make_borrowed(nullptr, 0, device) 的空 Memory。这避免了对零字节分配行为的设备依赖，也让上层逻辑更简单。

6.5 PinnedHostBuffer 的位置

虽然 Tensor 本身不直接依赖 pinned host buffer，但 allocator 模块里已经准备了 PinnedHostBuffer，通过 cudaHostAlloc 和 cudaFreeHost 管理 page-locked host memory。这通常是后续做高带宽 host-device staging 时的基础设施。

7. 边界条件与限制

这一层实现很克制，所以有不少能力是故意不做的。文档里把这些限制讲清楚，后面扩展时就不会误解当前行为。

7.1 当前已经明确支持的行为

  • borrowed host memory 包装成 Tensor，用于测试和外部缓冲区视图。
  • CUDA tensor 的 empty 和 zeros 分配。
  • 连续张量 reshape。
  • 单维 slice view。
  • 通过 data_as<T>() 直接取 typed pointer 给 kernel 和 cuBLAS 使用。

7.2 当前没有支持的行为

  • 没有 CPU allocator 实现，动态分配 CPU tensor 目前不是主路径。
  • 没有 transpose、permute、broadcast、advanced indexing。
  • 没有 copy-on-write，也没有 autograd。
  • 没有通用的非连续 reshape 推导。
  • 没有 dtype 自动转换或 device 自动迁移。

7.3 典型异常条件

场景 抛出条件 意义
负 shape size < 0 防止 numel、stride 推导失真。
shape/stride rank 不一致 sizes.size() != strides.size() 防止 view 元数据不完整。
reshape 非连续 tensor !is_contiguous() 明确拒绝复杂布局重解释。
reshape 改变 numel compute_numel(new_sizes) != numel() 保证 view 不越界。
slice 越界 dim 或 start/end 非法 防止 offset 落到无效区域。
CUDA allocator 分配 CPU device device.type != kCUDA 明确当前 allocator 只负责 GPU 路径。
现在这个 Tensor runtime 的定位非常像“推理后端内部张量描述符”，不是用户态张量库。它的约束越明确，上层代码越不容易误用。

工程化总结

Tensor 模块的价值不在“功能很多”，而在“语义足够小且稳定”。它把内存所有权、视图偏移、连续布局和 allocator 接口压缩成一套很小的协议，刚好够给 nano-vllm-cpp 里的 kernel、linear、attention、KV cache 和测试体系复用。
