1. 模块总览 Step 7

这一步不是再发明新的算子，而是把 Step 5 的线性层、Embedding、Attention、Sampler，和 Step 6 的参数加载机制，真正组装成一个能表达 Qwen3 参数树和前向路径的模型层级。

补齐 wrapper

Step 7 新增了 RMSNorm、RotaryEmbedding、SiluAndMul 三个模块封装。它们不是新 kernel，而是给已有 kernel_ops 一个稳定、可注册参数、可嵌入模型树的面向对象接口。

模型层级

新增 Qwen3Attention、Qwen3MLP、Qwen3DecoderLayer、Qwen3Model、Qwen3ForCausalLM，字段名与 Python 版参数树保持一致，方便 Step 6 的 loader 直接命中。

加载桥接

Qwen3ForCausalLM::register_parameters 和 load_weights 已经接上 Step 6。也就是说，Step 7 完成后，模型类本身就知道如何把 q_proj / k_proj / v_proj 和 gate_proj / up_proj 映射到 packed 参数。

验证方式

当前不是直接拿真实 Qwen3-0.6B 做全量 logits 对比，而是先用 tiny config 构造一个可手算的 1 层模型，验证前向、参数注册和 tie embeddings 行为，确保组装逻辑正确。

模型字段命名与 Python 版对齐 loader 可注册真实参数树 当前实现仍然偏最小工作版 真实端到端 Runner 留给 Step 8
Step 7 完成后的类关系 Qwen3ForCausalLM ├── Qwen3Model │ ├── VocabParallelEmbedding embed_tokens │ ├── vector<Qwen3DecoderLayer> layers │ │ ├── RMSNorm input_layernorm │ │ ├── Qwen3Attention self_attn │ │ │ ├── QKVParallelLinear qkv_proj │ │ │ ├── RowParallelLinear o_proj │ │ │ ├── RotaryEmbedding rotary_emb │ │ │ ├── Attention attn │ │ │ └── optional<RMSNorm> q_norm / k_norm │ │ ├── RMSNorm post_attention_layernorm │ │ └── Qwen3MLP mlp │ │ ├── MergedColumnParallelLinear gate_up_proj │ │ ├── SiluAndMul act_fn │ │ └── RowParallelLinear down_proj │ └── RMSNorm norm └── ParallelLMHead lm_head

2. 文件与类型地图

Step 7 实际跨了两层：一层是 wrapper，把 kernel 变成模块；另一层是 Qwen3 模型树，把这些模块拼成完整的参数结构与 forward 路径。

文件 角色 在 Step 7 中负责什么
include/layers/layernorm.h
src/layers/layernorm.cpp		 RMSNorm wrapper 封装单输入 RMSNorm、双输入 add+rms_norm，以及 weight_loader；构造时把权重初始化为全 1。
include/layers/rotary_embedding.h
src/layers/rotary_embedding.cpp		 RoPE wrapper 预计算 cos_sin_cache，并在前向里调用 rotary_embedding kernel。
include/layers/activation.h
src/layers/activation.cpp		 激活 wrapper 为 MLP 的 gate/up 输出分配结果 Tensor，再调用 silu_and_mul kernel。
include/models/qwen3.h
src/models/qwen3.cpp		 模型树 定义 Qwen3Attention / MLP / DecoderLayer / Model / ForCausalLM，并实现参数注册与 load_weights 入口。
tests/test_models.cpp 模型级测试 验证 tiny Qwen3 的前向输出、参数树注册、packed mapping 可见性、以及 tie embeddings 行为。

3. Step 7 补齐的 layer wrapper wrapper

Step 5 已经有 kernel 或函数级实现，但模型树还缺“可持有参数、可复用、可注册 loader”的对象层。Step 7 补的其实就是这一层。

RMSNorm

构造时分配一维 weight_，并根据 dtype 用 host 常量把它初始化成全 1。前向分成两条路径：

  • 单输入：直接调用 rms_norm(input, weight_, eps_)
  • 双输入：调用 add_rms_norm(input, residual, weight_, eps_)，同时返回 output 和新的 residual_out

这让 DecoderLayer 可以很自然地复用“第一层”和“后续层”的两种残差路径。

RotaryEmbedding

构造函数里就把 cos / sin cache 在 host 上算好，再一次性拷到 CUDA。前向时不再重新算频率，只做位置索引和旋转应用。

当前实现要求 rotary_dim == head_size 且 rotary_dim 为偶数，也就是说它针对当前 Qwen3 场景做了收敛，没有扩成更泛化的 partial rotary。

SiluAndMul

它几乎不持有状态，只做两件事：

  • 检查最后一维能不能被 2 整除
  • 分配一个最后一维减半的输出 Tensor，然后调用 silu_and_mul

这样 MLP 代码就可以直接写成 gate_up -> act_fn -> down_proj，不用每次手动分配输出。

// Step 7 不是再写一遍 kernel，而是把 Step 5 的算子收口成模块接口。
RMSNorm.forward(x)                 -> rms_norm(...)
RMSNorm.forward(x, residual)       -> add_rms_norm(...)
RotaryEmbedding.forward(pos, q, k) -> rotary_embedding(...)
SiluAndMul.forward(gate_up)        -> silu_and_mul(...)

4. Qwen3Attention attention

这是 Step 7 里信息最密的一层。它一方面要把 TP 下的 head 切分和 Step 5 的线性层接口接起来，另一方面要保持字段名和参数树命名与 Python 版一致。

4.1 构造函数做了哪些事

字段 来源 作用
num_heads_ / num_kv_heads_ 总 head 数除以 tp_size 表达当前 rank 的局部 head 数量。
head_dim_ config.head_dim 或 hidden / num_heads 与 Python 版一致，允许 HF config 显式提供。
q_size_ / kv_size_ 局部 head 数乘以 head_dim 后面切 qkv 时就按这个大小分块。
qkv_proj_ QKVParallelLinear 生成 packed 的 [q | k | v] 连续输出。
o_proj_ RowParallelLinear 把 attention 输出重新投回 hidden size。
rotary_emb_ RotaryEmbedding 持有 RoPE cache，并在前向应用到 q/k。
q_norm_ / k_norm_ 仅 attention_bias == false 时创建 贴近 Python 版的 Qwen3 分支：无 bias 时做 per-head RMSNorm。

4.2 forward 的实际执行顺序

hidden_states -> qkv_proj_.forward(...) -> qkv : [N, q_size + kv_size + kv_size] -> slice_last_dim_contiguous(q) -> slice_last_dim_contiguous(k) -> slice_last_dim_contiguous(v) -> reshape q : [N, num_heads_, head_dim_] -> reshape k : [N, num_kv_heads_, head_dim_] -> reshape v : [N, num_kv_heads_, head_dim_] -> if !qkv_bias_: q_norm, k_norm -> rotary_emb_.forward(positions, q, k) -> attn_.forward(q, k, v) -> reshape output : [N, q_size_] -> o_proj_.forward(...) -> attention output

4.3 为什么这里会出现 slice_last_dim_contiguous

这是当前 Step 7 最值得专门解释的一点。QKVParallelLinear 输出的是一个 packed 的连续 Tensor，但 Step 5 的 Tensor 运行时还没有提供“设备侧沿最后一维切片并直接得到连续张量”的高层操作。于是 Step 7 在模型层里加了一个内部 helper：

  • 先把整个输入 Tensor 拷回 host
  • 按最后一维范围做逐行 memcpy
  • 再把结果拷回目标设备 Tensor
这说明当前 Step 7 是“功能优先”的最小工作版。它保证 q/k/v 切分与 reshape 语义正确，但还不是最终性能路径。后续如果要做真正的高性能推理，这里应该被设备侧 contiguous narrow/copy 或更低层 view/copy primitive 替换。

4.4 参数注册树

Qwen3Attention::register_parameters 会把以下名字挂到注册表里：

  • ...self_attn.qkv_proj.weight
  • ...self_attn.qkv_proj.bias，仅当 qkv bias 开启
  • ...self_attn.o_proj.weight
  • ...self_attn.o_proj.bias，仅当 o_proj 带 bias
  • ...self_attn.q_norm.weight / k_norm.weight，仅当 qkv bias 关闭

这样 Step 6 loader 就可以直接把 HF 权重映射到这里，而不需要再猜模型内部结构。

5. Qwen3MLP 与 Qwen3DecoderLayer

这一节的重点不是“结构长什么样”，而是“Step 7 是如何把残差流和 MLP 压成当前已有算子接口的”。

5.1 Qwen3MLP

当前实现严格跟 Python 版对齐：

gate_up = gate_up_proj(x)
activated = act_fn.forward(gate_up)
output = down_proj(activated)

其中：

  • gate_up_proj_ 是 MergedColumnParallelLinear(hidden, [intermediate, intermediate])
  • act_fn_ 是上面补齐的 SiluAndMul
  • down_proj_ 是 RowParallelLinear(intermediate, hidden)
构造函数里明确校验了 hidden_act == "silu"。也就是说，这个实现是 Qwen3 定向的，不是“任意 MLP 激活都能套”的通用模块。

5.2 Qwen3DecoderLayer 的残差路径

这一层最容易被误读的地方，是 residual 不是每层都重新创建，而是沿着每层 forward 一直往后传。当前实现正是按 Python 版的 Pre-Norm 逻辑写的：

阶段 当前实现 含义
第一层进入时 residual 未定义，直接令 next_residual = hidden_states，然后对 hidden_states 做普通 RMSNorm 建立第一份残差基线。
后续层进入时 走 input_layernorm_.forward(hidden_states, residual) 调用 fused add_rms_norm，同时得到新的归一化输出和新的 residual。
attention 之后 再次走 post_attention_layernorm_.forward(hidden, next_residual) 把 attention 输出和残差重新融合，再进入 MLP。
返回值 {hidden_after_mlp, residual_after_post_attn} 供下一层继续复用残差流。
DecoderLayer.forward(positions, hidden_states, residual) if residual is undefined: residual = hidden_states hidden = input_layernorm(hidden_states) else: hidden, residual = input_layernorm(hidden_states, residual) hidden = self_attn(hidden) hidden, residual = post_attention_layernorm(hidden, residual) hidden = mlp(hidden) return hidden, residual

5.3 KV cache 注入点

Qwen3DecoderLayer::set_kv_cache 只是把工作转交给 self_attn_。这很重要，因为这说明 Step 7 已经把“每层 attention 需要独立 cache 视图”这个接口预留好了，但具体 cache 的整体分配与切片时机仍然属于 Step 8 的 Runner 责任。

6. Qwen3Model 与 Qwen3ForCausalLM

Step 7 最核心的交付物，不是某个单独类，而是完整的模型树已经成立：能接收 token 和 positions，能把输出交给 lm_head，能把参数树完整暴露给 loader。

6.1 Qwen3Model

当前 Qwen3Model 的 forward 很直白：

hidden_states = embed_tokens(input_ids)
residual = undefined
for layer in layers:
    hidden_states, residual = layer.forward(positions, hidden_states, residual)
if residual is defined:
    hidden_states = norm.forward(hidden_states, residual).first
else:
    hidden_states = norm.forward(hidden_states)
return hidden_states

这里的几个实现细节值得注意：

  • embed_tokens_ 与 norm_ 都由 Config::hf_config 初始化，不需要手写重复参数。
  • layers_ 用 vector<Qwen3DecoderLayer> 持有，并在构造函数里按 num_hidden_layers 循环 emplace。
  • 最终 norm 也优先走 fused 残差路径，这和 Python 版的 hidden_states, _ = norm(hidden_states, residual) 是对齐的。

6.2 Qwen3ForCausalLM

Qwen3ForCausalLM 只做三件关键事情：

  • 持有 model_ 与 lm_head_
  • 当 tie_word_embeddings 为真时，把 lm_head_ 的 weight 直接指向 embed_tokens_ 的 weight
  • 暴露 forward、compute_logits、register_parameters、load_weights 这四个对外接口
当前的权重共享不是“复制一份数据”，而是把 Tensor 句柄直接赋给 lm_head_。测试里验证的是两者 data() 指针相同，这符合“共享一块底层存储”的语义。

6.3 模型级接口边界

接口 当前职责 不负责什么
forward(input_ids, positions) 跑 token embedding 到 final norm 的主干前向 不做采样，不做 Scheduler/Runner 输入准备
compute_logits(hidden_states) 把 hidden states 送进 lm_head_ 不做 softmax / sampler
set_layer_kv_cache(layer_index, ...) 把单层 KV cache 句柄注入 attention 不负责整体 KV cache 分配
load_weights(path, allocator) 构建 registry 并调用 Step 6 loader 不做模型目录发现、Runner 初始化、真实端到端比对

7. 参数树注册、packed 映射与 load_weights loader bridge

Step 7 和 Step 6 的真正接口，不是某个单独的 weight_loader，而是“整个模型是否能把自己的参数树按正确名字注册出去”。这一节讲的就是这个桥。

7.1 packed_modules_mapping

Qwen3ForCausalLM::packed_modules_mapping() 当前返回固定 5 项映射：

HF 原始名片段 模型内部目标名 shard_id 含义
q_proj qkv_proj 0 写入 packed QKV 的 Q 区间
k_proj qkv_proj 1 写入 K 区间
v_proj qkv_proj 2 写入 V 区间
gate_proj gate_up_proj 0 写入 gate 区间
up_proj gate_up_proj 1 写入 up 区间

7.2 register_parameters 是怎么铺开整棵树的

Qwen3ForCausalLM::register_parameters ├── model_.register_parameters("model") │ ├── model.embed_tokens.weight │ ├── model.layers.0.input_layernorm.weight │ ├── model.layers.0.self_attn.qkv_proj.weight │ ├── model.layers.0.self_attn.o_proj.weight │ ├── model.layers.0.self_attn.q_norm.weight / k_norm.weight │ ├── model.layers.0.post_attention_layernorm.weight │ ├── model.layers.0.mlp.gate_up_proj.weight │ ├── model.layers.0.mlp.down_proj.weight │ ├── ... more layers ... │ └── model.norm.weight └── lm_head.weight

这意味着 Step 6 loader 不再需要一个“模型无关的外部参数表”，因为模型类自己就知道完整的注册逻辑。

7.3 load_weights 的入口非常薄

ParameterRegistry registry;
register_parameters(registry);
registry.load(path, packed_modules_mapping(), allocator, stream);

这段代码短，但信息密度很高：

  • 模型类自己负责创建并填充 registry
  • Step 6 仍然负责 SafeTensors 解析、packed 名称解析与 copy 规则
  • Step 7 负责把“参数树命名”和“每个参数落到哪个模块/哪个 shard”这件事变成显式代码

8. 测试怎么证明 Step 7 可用

当前测试不是为了证明“真实 Qwen3-0.6B 已完全还原”，而是为了证明 Step 7 作为组装层，前向结构、参数注册和权重共享语义都成立。

8.1 Tiny Qwen3 前向测试

Qwen3ForCausalLmForwardAndLogitsMatchReference 用一个极小配置手工构造模型：

  • hidden_size = 2
  • num_attention_heads = 1
  • num_hidden_layers = 1
  • intermediate_size = 2
  • vocab_size = 3

然后通过几组刻意简单的权重，让 attention 近似保持输入结构、让 MLP 权重全零，以便把期望输出压缩成“embedding 经过 RMSNorm 后，再乘 lm_head”的可手算形式。

测试设置 目的
qkv_proj.weight 设成分块 identity 风格 让 q / k / v 的语义容易追踪。
o_proj.weight 设成 identity 让 attention 输出不再被额外混合。
gate_up_proj 与 down_proj 全零 把 MLP 的影响消掉，专注验证主干组装逻辑。
所有 norm weight 设成全 1 让期望值退化成标准 RMSNorm，可直接手算。

8.2 参数树注册测试

Qwen3RegistersExpectedParametersAndPackedMappings 检查的是“模型类是否真的暴露了 loader 需要的参数路径”。它验证了：

  • model.embed_tokens.weight 存在
  • model.layers.0.self_attn.qkv_proj.weight 存在
  • model.layers.0.mlp.gate_up_proj.weight / down_proj.weight 存在
  • model.norm.weight 与 lm_head.weight 存在
  • packed_modules_mapping 返回 5 项，且第一项就是 q_proj -> qkv_proj, shard 0

8.3 权重共享测试

Qwen3TiesEmbeddingAndLmHeadWeightsWhenRequested 不看数值，只比较两个 Tensor 的 data() 指针是否相同。这个测试证明当 tie_word_embeddings = true 时，lm_head 确实共享 embed_tokens 的底层存储，而不是复制一份内容。

当前仓库在 Step 7 完成后，模型级测试 test_models 已接入 CMake，并且和现有 test_config / test_sequence / test_block_manager / test_scheduler / test_tensor / test_layers / test_loader 一起通过了完整 ctest。

9. 当前限制与下一步

这一节很重要。Step 7 已经完成“模型树成立”，但并不等于整个推理引擎已经端到端闭环。下面这些边界必须明确写清楚。

  • 当前实现仍然是最小工作版，默认工作路径主要围绕当前仓库已有的 float32 导向算子栈。
  • Qwen3Attention 中的 slice_last_dim_contiguous 会走 host-side materialization，用来保证 q/k/v 切片语义正确，但这不是最终高性能路径。
  • RotaryEmbedding 当前只支持 rotary_dim == head_size 的全量 rotary，不支持更复杂的 partial rotary / rope_scaling 分支。
  • Step 7 已经有 set_layer_kv_cache 接口，但 KV cache 的整体分配、每层切片注入、prefill/decode 输入准备仍属于 Step 8 的 ModelRunner 工作。
  • load_weights 已能调用 Step 6 loader，但“真实 Qwen3-0.6B 权重 + 与 Python 版 logits 对齐到 < 1e-3”这个验证目标还没有在当前文档对应的实现里完成。
  • 多卡 all-reduce / gather 已在底层层模块语义上预留，但真正的通信执行仍属于后续多卡阶段，不是 Step 7 的交付物。
Step 7 之后的真实边界 已经完成： kernel wrapper -> 模块接口 Qwen3 模型树 -> forward / compute_logits 参数注册树 -> Step 6 loader bridge tie embeddings -> 共享 Tensor 存储 tiny config 模型测试 -> 结构正确性验证 尚未完成： ModelRunner 输入准备与 KV cache 全局挂载 真实模型目录加载后与 Python 版 logits 对齐验证 qkv 分片的高性能设备侧 contiguous 路径 真正的多卡通信调用
Step 7 文档基于当前仓库实现生成，强调“代码已做成什么”而不是“架构最终想做什么”。
