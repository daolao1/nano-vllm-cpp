nano-vllm-cpp Scheduler 模块讲解

基于 include/engine/scheduler.h、src/engine/scheduler.cpp、tests/test_scheduler.cpp 的实现解读。目标是把 Scheduler 在 continuous batching 中扮演的角色讲清楚：什么时候做 prefill，什么时候做 decode，什么时候触发 LIFO 抢占，以及它如何和 BlockManager 协作。

1. 模块总览

一句话总结：Scheduler 负责“决定这一轮哪些 Sequence 能上 GPU”，但它本身不做模型计算，也不直接管理 KV 数据内容。它做的是批次选择、资源协调和状态迁移。

Prefill 优先

waiting 队列里只要还有新请求，Scheduler 就优先尝试把它们以 prefill 方式拉进本轮执行，而不是先继续旧请求的 decode。

连续批处理

prefill 结束后，已进入 running 的请求会在后续 step 中持续参与 decode，直到 finished 或被 preempt。

资源仲裁

Scheduler 不自己分配 block，而是通过 BlockManager 的 can_allocate、allocate、can_append、may_append 来判断和执行 KV 页操作。

状态机驱动

WAITING、RUNNING、FINISHED 三种状态由 Scheduler 维护，Sequence 自身只是状态载体，不主动调度自己。

用户 add(seq) │ ▼ waiting_ ────── schedule() ──────► running_ ────── postprocess() ──────► FINISHED │ │ │ │ │ │ │ └─ 命中 EOS / 达到 max_tokens │ │ │ │ └─ prefill 优先 └─ decode 轮转执行 │ └──── preempt(seq) ◄──── 显存页不足时 LIFO 抢占
调度单位是 Sequence 资源单位是 Block step 返回的是本轮待执行批次 ModelRunner 只消费调度结果

2. 数据结构与所有权

Scheduler 真正容易踩坑的不是算法，而是 C++ 里的对象生命周期。实现里专门为这个问题做了稳定地址设计。

int max_num_seqs_;
int max_num_batched_tokens_;
int eos_;
BlockManager block_manager_;

std::list<Sequence> sequences_;
std::deque<Sequence*> waiting_;
std::deque<Sequence*> running_;
字段 职责 为什么这样设计
max_num_seqs_ 单轮最多调度多少条序列 限制 batch 里的 sequence 数，避免 decode 批次无限增长。
max_num_batched_tokens_ prefill 阶段的 token 上限 prefill 是按 token 数限制的，因为 prompt 长度差异很大。
eos_ 终止 token postprocess 需要据此判断一条序列是否完成。
block_manager_ KV 页资源管理器 Scheduler 只做策略，不直接碰物理页细节。
sequences_ 真正拥有 Sequence 对象 std::list 节点地址稳定，waiting_ / running_ 里保存的指针不会因为 push/pop 失效。
waiting_ 等待被调度的序列 用指针队列实现“入队、出队、抢占回插”。
running_ 已经分到 block、后续可 decode 的序列 decode 阶段要从队首轮转、从队尾抢占，deque 很合适。
如果直接让 waiting_ 自己存 Sequence 对象、running_ 再存指针，那么对象在移动、弹出、回插时很容易导致指针悬空。当前实现故意把“对象拥有权”和“调度队列”拆开，就是为了避免这个问题。

2.1 add 的语义

void Scheduler::add(Sequence seq) {
    sequences_.push_back(std::move(seq));
    waiting_.push_back(&sequences_.back());
}

这一步做了两件事：

  • Sequence 对象被正式纳入 Scheduler 生命周期管理。
  • waiting_ 只保存指向该对象的稳定指针，后续 schedule / preempt / postprocess 都围绕这根指针工作。

3. schedule 两阶段流程

schedule 的核心思想是“先试图消化新请求，再轮到老请求继续 decode”。这就是 continuous batching 中 Prefill 优先的策略来源。

std::pair<std::vector<Sequence*>, bool> schedule();
schedule(): scheduled = [] Phase 1: Prefill while waiting_ 非空: 检查 token budget 检查 block_manager_.can_allocate() allocate(seq) seq.status = RUNNING 从 waiting_ 移到 running_ 加入 scheduled if scheduled 非空: return (scheduled, true) Phase 2: Decode while running_ 非空: 从 running_ 头部取 seq 若 can_append 失败: 抢占 running_ 尾部序列，必要时连自己也抢占 否则: may_append(seq) 加入 scheduled return (scheduled, false)

3.1 Prefill 阶段 新请求优先

prefill 部分使用两个约束决定还能不能继续装请求：

  • num_seqs < max_num_seqs_：不能超过 sequence 数上限。
  • num_batched_tokens + seq->len() > max_num_batched_tokens_：不能超过 token 总预算。
  • block_manager_.can_allocate(*seq)：不能超过当前剩余 block 资源。
num_batched_tokens += seq->len() - seq->num_cached_tokens;

这里很关键。prefill 预算统计的是“本轮真正要算的 token 数”，不是原始 prompt 长度。若 prefix cache 命中了一部分，真正的 GPU 工作量会更小，Scheduler 必须扣掉这部分缓存前缀。

3.2 为什么 prefill 一旦有结果就直接返回

因为当前设计把一轮 step 明确区分为“prefill 轮”或“decode 轮”。如果 waiting_ 里还有新请求，就优先把这些请求推进系统，让它们尽快完成首次建图和首 token 生成，而不是让老请求持续霸占 decode 机会。

这意味着 decode 不是绝对公平的。只要新请求持续到达，老请求的 decode 可能被延后。这是 Prefill 优先策略的直接代价，也是吞吐与首 token 延迟之间的权衡。

3.3 Decode 阶段 running 轮转

当 waiting_ 已经空，或者 waiting_ 队首因为资源限制暂时装不下时，Scheduler 才会进入 decode 路径。

  • 从 running_ 队首取出一个 seq。
  • 调用 block_manager_.can_append(*seq) 判断新 token 追加后是否需要新 block、且当前资源是否足够。
  • 如果能 append，就调用 may_append 维护 block_table 和 hash 注册状态。
  • 成功调度的序列会在函数末尾按原顺序重新塞回 running_ 前部，保持后续轮转一致性。
running_ 初始: [A, B, C] 依次从前往后取出: 取 A -> 可 decode -> scheduled = [A] 取 B -> 可 decode -> scheduled = [A, B] 取 C -> 若资源不足, 可能被抢占 最后把 scheduled 逆序 push_front: push_front(B), push_front(A) 结果 running_ 仍保持 [A, B, ...剩余序列] 这样下一轮 decode 的遍历顺序不会被打乱。

4. 抢占机制 LIFO preempt

抢占只发生在 decode 阶段，而且使用的是 LIFO 策略：优先踢掉最晚进入 running 的序列。

while (!block_manager_.can_append(*seq)) {
    if (!running_.empty()) {
        Sequence* victim = running_.back();
        running_.pop_back();
        preempt(victim);
    } else {
        preempt(seq);
        break;
    }
}

4.1 为什么是 LIFO

  • 越晚进入 running 的请求，通常已经消耗的 decode 步数更少，抢占它的历史损失更小。
  • 较早进入 running 的老请求更接近完成，保留它们有助于缩短排队中的整体尾延迟。
  • 实现简单，且和 Python 版 nano-vllm 的策略对齐。

4.2 preempt 实际做了什么

void Scheduler::preempt(Sequence* seq) {
    seq->status = SequenceStatus::WAITING;
    block_manager_.deallocate(*seq);
    waiting_.push_front(seq);
}

注意这里不是“丢弃请求”，而是“退回等待态重新参与后续 prefill”。具体效果是：

  • 状态回到 WAITING。
  • 当前占用的 block 全部释放。
  • 重新插回 waiting_ 队首，后面有机会再次走 prefill。
显存页不够时: running_ = [A, B, C, D] 当前处理 A, 但 A append 需要新页 策略: 先抢占 D 如果还不够, 再抢占 C ... 最坏情况连 A 自己也抢占 被抢占的序列: status = WAITING block_table 清空 回到 waiting_ 队首

4.3 “连自己都放不下”是什么意思

如果 running_ 里已经没有别的序列可踢，但当前 seq 自己 append 一个 token 依然无法满足 block 需求，那它也只能被 preempt。此时本轮不会调度它，schedule 会继续尝试其它可能的序列；如果一个都没有，就断言失败，这表示当前系统设计假定“进入 decode 时至少应有一个序列可被成功安排”。

5. postprocess 生命周期管理

postprocess 发生在 ModelRunner 返回采样结果之后。Scheduler 在这里把“模型输出 token”转回“系统状态变化”。

void postprocess(const std::vector<Sequence*>& seqs,
                 const std::vector<int32_t>& token_ids);
for each (seq, token_id): 1. seq.append_token(token_id) 2. 判断是否结束: - 命中 EOS 且 !ignore_eos - 或 num_completion_tokens() == max_tokens 3. 若结束: - status = FINISHED - block_manager_.deallocate(*seq) - 从 running_ 删除

5.1 为什么在 postprocess 才 append_token

因为 schedule 只负责“安排这一轮谁去跑”，它不知道模型真正会生成什么 token。只有当采样结果回来后，Sequence 的 token_ids、last_token、num_tokens 才能被正式更新。

5.2 结束条件

条件 含义 后果
!ignore_eos && token_id == eos_ 模型显式生成结束符 序列转 FINISHED，释放全部 block。
num_completion_tokens() == max_tokens 达到用户设定的最大生成长度 即使没出 EOS，也强制结束。

5.3 为什么 finished 后还保留 Sequence 对象

当前实现并没有把 FINISHED 对象从 sequences_ 中抹掉，只是把它从 running_ 移除并释放 block。这样做的好处是：之前返回给外部的 Sequence* 仍然有效，调用方后续仍可读取最终 token_ids、seq_id 等结果。

6. 与 BlockManager 的协作边界

Scheduler 和 BlockManager 分工非常清晰：一个做“策略”，一个做“资源”。读懂它们的接口边界，后面接 ModelRunner 会顺很多。

阶段 Scheduler 决策 BlockManager 动作
add 把请求纳入 waiting_ 无
prefill 检查 是否还有 token/seq budget can_allocate(seq)
prefill 分配 把序列从 waiting_ 移入 running_ allocate(seq)，建立 block_table 并处理 prefix cache
decode 检查 决定是否继续当前 seq，或先 preempt 别人 can_append(seq)
decode 维护 把可执行序列加入本轮 scheduled may_append(seq)，必要时追加 block 或注册 hash
抢占 把序列打回 waiting_ deallocate(seq)
完成 标记 FINISHED 并移出 running_ deallocate(seq)
Scheduler 自己不认识 block_id、hash、prefix chain 这些底层概念。它只通过 BlockManager 暴露出来的资源接口做判断。这样分层之后，未来即使换别的 cache 管理策略，调度逻辑也不需要大改。

7. 关键不变量与工程实现

这个模块虽然代码不长，但有几个不变量必须一直成立，否则后面接 ModelRunner 时会出现非常难查的 bug。

7.1 schedule 返回值语义固定

  • 返回的 bool 为 true 时，表示本轮是 prefill 批次。
  • 返回的 bool 为 false 时，表示本轮是 decode 批次。
  • 一个批次不会混合 prefill 和 decode 序列。

7.2 waiting_ / running_ 不应重复持有同一条活跃序列

同一条未完成序列在任意时刻只能处于 WAITING 或 RUNNING 其中之一。抢占的实现通过“先从 running_ 弹出，再 push 到 waiting_”保证了这一点。

7.3 running_ 中的序列必须已有 block

prefill 成功后才会把序列从 waiting_ 移入 running_。因此 running_ 中的序列应满足：

  • status == RUNNING
  • block_table 已经建立
  • decode 路径可以直接调用 can_append / may_append

7.4 FINISHED 序列不再参与调度

postprocess 一旦把某条序列标记为 FINISHED，就会同步从 running_ 删除并释放 block。因此 is_finished() 可以简单定义为 waiting_.empty() && running_.empty()。

7.5 当前实现中的一个显式假设

schedule() 开头有 assert(!is_finished())，decode 末尾有 assert(!scheduled_seqs.empty())。这说明当前实现假设调用方不会在“完全没有可调度请求”时继续调用，也假设 decode 路径一旦进入，总能找到至少一条能成功安排的序列。

如果未来要把这个模块用于更复杂的 admission control，可能需要把这些 assert 改成显式错误返回，否则某些极端资源场景会直接崩溃而不是优雅降级。

8. 测试覆盖与阅读建议

当前测试没有追求穷尽所有调度排列，但已经覆盖了这个阶段最关键的 3 个行为断言。

测试名 验证点 你应该关注什么
PrefillIsPrioritizedOverDecode 有新请求进来时，本轮仍优先返回 prefill 说明调度策略是“prefill 抢优先级”，而不是让 running 里的老请求一直 decode。
DecodePreemptsLatestRunningSequenceWhenOutOfBlocks decode 扩页失败时，会抢占 running 队尾序列 重点看 LIFO preempt 是否真把被抢占序列打回 waiting，并清空 block_table。
PostprocessMarksFinishedAndRecyclesBlocks 命中 EOS 后能正确完成并释放 block 重点看 FINISHED 后能否再次接受新请求，证明资源确实回收成功。

8.1 阅读顺序建议

  • 先看 schedule()，把 prefill / decode 两阶段的大框架建立起来。
  • 再看 preempt()，理解显存不足时系统如何回退。
  • 最后看 postprocess()，把“模型输出”如何转成“状态迁移”串起来。
