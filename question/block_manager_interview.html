AI Infra 面试题

面试官身份：AI Infra 资深工程师。不了解你的项目细节，围绕“你实现了一个 LLM 推理引擎里的页式 KV Cache 管理模块”提问。题目使用通用领域语言，参考答案再结合你当前的 BlockManager 实现展开。

一、基础认知（确认你理解页式 KV Cache 和前缀复用的核心原理）

1 你说你做了页式 KV Cache 管理，为什么不直接给每条请求分一大段连续显存，而要切成固定大小的页？ LLM ▶
面试官意图：考察是否理解 PagedAttention 背后的资源管理动机，而不是只会实现一个 free list
参考答案

连续分配的主要问题是浪费和不可伸缩。如果每条请求都按最大长度预留一整段 KV Cache，那么短请求也会占掉大量无效空间，显存利用率很差；而且序列在 decode 中逐 token 增长，连续扩容会涉及拷贝或复杂的内存搬移。

固定大小分页的好处有三点：

  • 按需分配：序列增长多少 token 就拿多少页，尾部最多只浪费一个页的剩余空间
  • 便于回收和共享：前缀命中时多个序列可以共享同一个物理页
  • 便于调度：调度器只需要判断当前还剩多少页，而不是管理大段连续区间

我当前实现里就是把 KV Cache 资源抽象成固定大小的 block，由 Sequence 维护逻辑 block 列表，再由 BlockManager 决定这些逻辑块映射到哪些物理页。

页大小设大还是设小有什么 trade-off？
2 在这种设计里，为什么要把“逻辑 token 序列”和“物理页分配”分成两层，而不是让执行器直接按 token 下标找地址？ 工程 ▶
面试官意图：考察 Sequence 和 BlockManager 的职责边界是否清楚
参考答案

因为 token 的逻辑顺序和 KV 的物理落点不是一回事。序列对上层来说是连续的，但页式分配下物理页是离散的。如果不显式分层，下游每个模块都得知道如何从 token 下标推回物理页，耦合会很重。

我当前的分层是：

  • Sequence：负责完整 token 序列、逻辑切块、缓存前缀长度、物理页映射表
  • BlockManager：负责全局页池、分配/回收、hash 索引、引用计数
  • 执行器：消费 block_table 和 slot_mapping 去准备 attention 输入

这样 Sequence 是控制平面对象，BlockManager 是资源管理器，执行器只读这些元数据，不负责维护它们。

如果把 block table 挪到 BlockManager 里做外部索引，会带来什么问题？
3 你怎么向别人解释“前缀复用”这件事？为什么它复用的是连续前缀，而不是某几个恰好内容相同的页？ LLM ▶
面试官意图：确认候选人真正理解 prefix cache 的语义，而不是只背了一个哈希表
参考答案

前缀复用的本质是：如果两条请求在开头一段上下文完全相同，那么这一段上下文对应的 KV 就不需要重复计算。因为 attention 里的 K/V 只由历史 token 决定，相同历史就会产生相同 KV。

但它必须是连续前缀，不能跳着复用。原因是 attention 的上下文是有顺序的，如果中间某一页不一致，后面的页即使单独 token 内容一样，它对应的“前文历史”也已经不同了，不能再被视作同一段上下文。

我在实现里用 hash chain 强化这个约束：一旦某一页 miss，后面全部按 miss 处理，不会出现“中间断掉又继续命中”的情况。

如果想做到比“整页连续前缀”更细粒度的复用，你会想到什么数据结构？
4 为什么这类前缀缓存通常只缓存完整页，而不会急着把最后一个残缺页也注册成可复用缓存？ LLM ▶
面试官意图：考察对“稳定缓存单元”和 decode 增长语义的理解
参考答案

因为残缺页的内容还没稳定。decode 阶段后续 token 继续追加时，最后一页会不断变化；如果过早把它注册到缓存索引里，后面每次追加都要更新 hash 和索引，管理复杂度明显增加。

主流工程实现通常选择“只缓存完整页”，代价是最多损失一个页粒度的复用精度，但换来简单、稳定和易验证。

我当前实现也是这样：只有完整 block 才计算 hash；最后一个残缺 block 会把 hash 设成无效值，所以 prefix cache 只能命中到最后一个完整块。

如果一个请求已经结束，最后那个残缺页还要不要想办法也缓存下来？
5 哈希链在这里到底解决了什么问题？如果每个页单独做哈希，不也能查重吗？ 数学 ▶
面试官意图：确认候选人理解 hash chain 的必要性，而不是把它当成普通 hash table
参考答案

单页哈希只能回答“这一个页内容是不是一样”，但回答不了“到这一页为止的整段前缀是不是一样”。哈希链就是把前缀历史编码进当前页。

h_i = Hash(h_{i-1}, tokens_i)

这样当前页的 hash 不只依赖当前页 token，还依赖前一个页的 hash，也就间接依赖整个前缀。结果就是：

  • 如果前缀不同，即使某一页 token 内容恰好相同，hash 也不同
  • 只有当“从开头到这一页”为止都相同，当前 hash 才会相同

我当前实现用 xxhash64 做这个链式哈希，完整 block 才会进入链。

如果改成 Merkle tree 或 trie，这个问题会怎么变化？
6 页式管理器产出的页表，最后是怎么被模型执行路径消费的？它为什么不只是调度器内部的 bookkeeping？ GPU ▶
面试官意图：把元数据设计和后续 attention 执行路径真正连起来
参考答案

页表不是只给调度器看的，它会一路传到输入准备和 attention kernel。执行链路通常是：

  1. Sequence 保存逻辑 block 到物理 block 的映射
  2. 输入准备阶段根据这份映射生成 block_tables 和 slot_mapping
  3. attention kernel 根据 block table 去读历史 KV，根据 slot mapping 去写新 KV

所以 block_table 是把“控制平面上的页分配”转换成“执行平面上的访存地址”的桥梁。没有它，prefix cache 命中了哪些历史页、suffix 该写到哪些物理 slot，都没法准确表达。

如果只命中到倒数第二个完整页，最后一个残缺页没命中，输入准备应该怎么切 query 和 key 长度？

二、工程设计（页池、索引和生命周期管理）

7 你会给“一个物理页”设计哪些元数据？每个字段是为了解决什么问题？ 工程 ▶
面试官意图：考察抽象能力，看候选人能不能把页对象的职责说清楚
参考答案

我当前的物理页元数据主要有四类：

  • 页号：标识这是全局页池里的第几个 block
  • 引用计数：支持多个请求共享同一前缀页
  • hash：作为 prefix cache 的快速索引 key
  • token 内容：用于哈希命中后的二次校验，防止 collision 导致误复用

注意物理页对象不需要直接持有所有层的显存指针；在更完整的系统里，页号通常就足够让执行器定位对应层上的 KV 切片。

如果后面想进一步压缩元数据开销，哪个字段你最先考虑优化？
8 为什么需要引用计数？用一个简单的“是否占用”标记不行吗？ 工程 ▶
面试官意图：考察是否真的把共享前缀场景考虑进去了
参考答案

布尔占用标记只适合“一个资源同一时刻最多一个 owner”的场景，但 prefix cache 命中后，同一个物理页可能被多条序列共享。

典型场景是两条请求有相同的 prompt 前缀。第二条请求命中前缀后，不会复制一份 KV，而是直接复用第一条请求对应的物理页。这时释放任何一条请求都不能把这页回收，必须等所有引用都结束。

所以我当前实现里页对象维护 ref_count，只有降到 0 才会回到 free 队列。

如果未来要做并发分配，这个引用计数要不要改成原子？
9 一个页被 free 之后，什么叫“真正删除”？是在 free 的瞬间就擦掉，还是等下次复用时再覆盖？ 工程 ▶
面试官意图：考察候选人对资源回收和缓存保留语义的区分
参考答案

在这类实现里，free 通常表示“从活跃引用里移除”，不一定表示“立刻擦除内容”。更细地说有两层状态：

  • 活跃使用：还有序列引用它，不能回收
  • free 但仍 cache-resident：没人引用了，但内容和 hash 还在，后续相同前缀可能重新命中它

我当前实现里就是这样：deallocate 只是在 ref_count 归零时把页号放回 free 队列，并没有立刻清掉 hash 索引；真正 reset 元数据是在这个页号被重新分配给新内容时发生的。

这种“free 但仍保留旧内容”的设计，风险点在哪里？
10 只凭 hash 命中就直接复用一个页，你觉得安全吗？如果两个不同内容碰巧 hash 一样怎么办？ 数学 ▶
面试官意图：确认候选人不会为了省一点比较就牺牲正确性
参考答案

只比 hash 不够安全。正确做法是：

  1. 先用 hash 快速定位候选页
  2. 再用页内 token 内容做最终确认

这样 hash collision 最多让命中率变差，不会让错误 KV 被复用。我的实现里就是这么做的：哈希表只给出一个候选 block_id，随后还会比较保存下来的 token_ids 是否完全一致，不一致就按 miss 处理。

之所以这个额外比较不算太慢，是因为比较范围只是一页固定大小的 token，不是把所有历史请求整段扫一遍。

如果后面页大小很大，或者担心 hash collision 导致命中率下降，你会怎么改索引结构？
11 在 decode 阶段，为什么要区分“最后一页刚好填满”和“新 token 刚跨到下一页”这两个瞬间？ LLM ▶
面试官意图：考察是否理解 decode 下页状态变化的边界条件
参考答案

因为这两个瞬间对应的是完全不同的维护动作：

  • 刚好填满：当前残缺页从“不可缓存”变成“稳定可缓存”，应该注册 hash
  • 刚跨页：说明上一页已经满了，新的 token 需要一个新物理页

如果不把这两个边界拆开，页的 hash 注册时机和扩容时机就会混在一起，很容易出错。我的实现里 may_append 就是按余数的不同分支处理：一个分支负责新开页，一个分支负责把刚填满的页提升为可复用缓存。

如果后面支持 speculative decoding，一次接受多个 token，这个接口哪里会首先失效？
12 当页池资源不足时，你更倾向于扩容、等待还是抢占？怎么取舍？ 工程 ▶
面试官意图：考察调度策略和资源约束意识，而不是把问题推给显存自动扩容
参考答案

在线推理里通常不会在运行时动态扩容 GPU KV Cache，因为显存池一般在启动时就按预算一次性申请完了。运行期更常见的是：

  • prefill 放不下：新请求先留在 waiting 队列里，等页释放
  • decode 追加放不下：抢占一部分 running 序列，释放其页资源
  • 极端场景：如果单条请求本身就放不下，需要准入时拒绝或截断

我当前实现就是固定页池 + backpressure/preempt 的思路，没有做运行期扩容。

如果要支持 CPU swap 或 NVMe offload，你觉得 BlockManager 的职责要不要扩展？

三、实战与扩展（把当前实现和更复杂的生产问题连接起来）

13 如果现在所有页都被占用了，新来的请求放不进去，你的系统会怎么处理？ 工程 ▶
面试官意图：确认候选人把 BlockManager 和调度器的协作关系讲清楚了
参考答案

在固定页池设计下，新请求不会硬塞进去，也不会临时扩容。更合理的流程是：

  1. prefill 阶段先做容量检查
  2. 如果当前空闲页不够，新请求停留在 waiting 队列
  3. 系统继续服务已经 running 的序列
  4. 等旧请求结束释放页，或者 decode 阶段通过 preempt 腾出页，再重新尝试调度

我当前实现里的 can_allocate 是比较保守的，它按整条序列所需页数来估算，不提前把潜在的 prefix cache 命中节省量算进去。这保证简单和安全，但会牺牲一点可调度性。

如果你想把这一步做得更激进一些，会如何在调度前预估“实际新增页数”？
14 一个已经 free 的页，如果旧内容还留在显存里，后续请求还能不能重新命中它？ LLM ▶
面试官意图：考察对“释放活跃引用”和“驱逐缓存内容”两个概念的区分
参考答案

可以，只要这页还没被新内容覆盖，而且 hash 和 token 内容仍然匹配。free 只表示当前没有活跃请求引用它，不代表它的缓存价值已经消失。

我当前实现里就是这种语义：页 free 后如果以后又有相同前缀的请求到来，hash 表还能把它找出来；如果该页目前不在 used 集合，就把同一个页号重新激活。只有当它被重新分配给别的内容时，旧元数据才会 reset 并失效。

这种做法和传统 LRU 缓存相比，少了哪一层能力？
15 多卡张量并行时，页号空间应该怎么定义？是每层局部编号，还是全局编号？ GPU ▶
面试官意图：看候选人是否把页号、层维度、rank 边界想清楚了
参考答案

更自然的做法是：在单个 rank 的 KV cache 池内，页号是全局编号；层维度由 KV cache 张量的另一维来表达，不需要每层再维护一套独立页号。

这样一个逻辑 block 对应的是“每一层上同编号的一组页”。Sequence 里只需要一份 block_table，执行层再通过当前层号和 block_id 去定位本层的 K/V 存储。

如果是 tensor parallel，多卡之间各自有本 rank 的 KV cache 池，所以页号通常是 per-rank global，而不是整个集群唯一。

如果不同 rank 的可用页数不一样，这套设计会遇到什么麻烦？
16 如果后面要支持 speculative decoding，一轮可能确认多个 token，这套页管理接口哪些地方要先改？ LLM ▶
面试官意图：考察候选人能不能识别当前实现里的“每轮只增 1 token”假设
参考答案

当前这类简化实现通常默认 decode 每轮每序列只增 1 个 token，所以几个接口都隐含了单步假设：

  • Sequence 的追加接口只接受 1 个 token
  • 页池只判断“最多会不会多 1 个新页”
  • 刚填满页和刚跨页的分支只覆盖单 token 边界

如果支持 speculative decoding，接口应该改成接受变长 token 列表：追加接口要支持多 token，can_append 和 may_append 要按“本轮确认 K 个 token 需要几页”来计算，还要能一次注册多个刚填满的完整页。

如果本轮只确认了 draft 的前 2 个 token，后 3 个被拒了，页状态回滚怎么设计？
17 这种页式缓存管理器，你觉得单元测试最少该覆盖哪些行为，才能比较有把握？ 工程 ▶
面试官意图：考察是否有把资源管理模块当成“行为规格”来测，而不是只测 happy path
参考答案

我会至少覆盖这几类：

  • 哈希行为：同输入稳定、不同输入不同、prefix 不同导致 hash chain 不同
  • 分配/释放：单页、多页、页池满时拒绝分配
  • 前缀命中：共享前缀会复用页，不共享前缀不会误命中
  • 引用计数：共享页只在最后一个引用释放后才真正回收
  • decode 边界：未满页追加、跨页追加、刚填满页注册 hash
  • 边界输入：单 token 序列、残缺页、不完整尾页不会被缓存

我当前的测试文件基本也就是沿着这些行为写的，尤其把 prefix hit、deallocate 回收和 decode 边界都单独拆了测试。

如果你再补一轮测试，会优先加随机化 fuzz 还是更细的并发测试？为什么？
18 回看你现在这个页管理模块，如果让你自己挑一个“先保证正确、以后再优化”的点，你会选哪里？ 工程 ▶
面试官意图：看候选人是否能客观评价当前实现的保守点和下一步演进方向
参考答案

我会先提三个现在刻意保守的点，再说明取舍：

  • can_allocate 比较保守：按整条序列所需页数估算，没有提前扣掉潜在 prefix cache 命中带来的节省
  • block(i) 返回副本：实现简单安全，但有额外拷贝，后面可改成 span 之类的视图
  • 没有独立 eviction 策略：free 但可复用的页依赖“何时被覆盖”自然失效，还没做 LRU/TTL 这类缓存治理

这些都是“第 1 步先把正确性和接口语义打牢”的取舍。等调度器、执行器、压测都接起来之后，再看热点决定优化优先级会更稳。

如果只能做一个优化，你会优先改哪一项？为什么不是另外两项？
