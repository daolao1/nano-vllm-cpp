AI Infra 面试题

面试官身份：AI Infra 资深工程师。不了解你的项目细节，围绕“你实现了一个 LLM 推理引擎的调度器”提问。

一、模块定位（确认你理解 Scheduler 不是简单队列）

1 在一个 LLM 推理引擎里，调度器的职责到底是什么？为什么不能只是维护一个请求队列然后顺序执行？ 工程 ▶
面试官意图：判断你是否把调度器理解成“队列管理器”，还是理解成吞吐、时延和显存约束下的决策中心
参考答案

调度器的核心职责不是“存请求”，而是在每一轮 step 里决定哪些序列进入执行，以及它们以什么模式执行。

  • 批次形成：从 waiting/running 中挑选本轮要跑的序列
  • 模式选择：区分 prefill 还是 decode
  • 资源协调：和 BlockManager 协作判断是否有足够的 KV block
  • 生命周期推进：在 postprocess 阶段推进序列状态，必要时回收资源

如果只是顺序执行请求，吞吐会很差，也无法支持 continuous batching、prefix cache、抢占和显存回收。

我当前实现里的 Scheduler 直接持有 waiting/running 两个队列，并在 schedule() 里做两阶段调度，而不是仅做 FIFO 出队。

如果系统目标从吞吐优先改成 tail latency 优先，你觉得调度器要改哪些策略？
2 为什么调度器必须知道 KV Cache 的资源状态？把“排队”和“显存管理”完全拆开不行吗？ GPU ▶
面试官意图：考察你是否理解推理调度和显存分配是强耦合关系
参考答案

在 LLM 推理里，请求能不能执行不是只看“前面排了几个”，而是看本轮是否能拿到对应的 KV Cache 资源。

Prefill 要先为序列建立 block table，decode 则要判断追加一个 token 后是否需要新 block。所以调度器必须在调度决策时查询：

  • can_allocate()：prefill 前能否为整条序列建立初始 block
  • can_append()：decode 前能否承受潜在的新 block 需求
  • allocate()/may_append()/deallocate()：执行前后更新资源状态

如果把排队和显存管理完全拆开，调度器就可能选出一批根本无法执行的序列，最后还得二次回滚，系统复杂度会更高。

这层耦合会不会导致调度器职责过重？你会怎么控制接口边界？
3 为什么 prefill 和 decode 要分成两套调度逻辑，而不是统一成“每轮挑一批 token”这么简单？ LLM ▶
面试官意图：确认你理解 prefill 和 decode 的计算形态、资源消耗和输入组织方式不同
参考答案

因为两者的工作负载完全不同。

  • Prefill：通常一次处理整段 prompt，输入 token 数多，可能命中 prefix cache，主要关注总 token 数
  • Decode：每轮每序列通常只处理 1 个 token，但会持续很多轮，主要关注活动序列数和 append block 的可行性

这会影响 batching 方式、BlockManager 交互方式、甚至 ModelRunner 的输入准备。我的实现里 schedule() 先尝试 prefill，只有本轮没有任何 prefill 可做时才进入 decode 路径。

如果以后支持 chunked prefill，你会如何重构这两个阶段的边界？
4 为什么很多系统会采用“有新请求就优先 prefill”的策略？这样不会拖慢已经在 decode 的请求吗？ LLM ▶
面试官意图：看你是否理解 prefill 优先背后的吞吐和用户感知时延权衡
参考答案

prefill 优先的原因通常是：新请求只有完成 prefill 之后才有资格进入稳定 decode。如果始终让 decode 占满系统，新请求首 token latency 会非常差。

我当前实现采用的就是这个策略：只要 waiting 里能调度出至少一个 prefill 序列，就直接返回 prefill batch，不再继续 decode。

它确实可能让 decode 序列多等一轮，但换来的是更好的新请求接入能力和更稳定的整体吞吐。这个 tradeoff 很典型。

如果发现 decode 被饿死，你会改成时间片、配额，还是双队列配比？为什么？
5 为什么调度器通常同时需要“最大序列数”和“最大 token 数”两个上限？只保留一个不够吗？ 工程 ▶
面试官意图：考察你是否理解 batch 维度和 token 维度对计算成本的不同影响
参考答案

不够，因为这两个维度约束的不是同一件事。

  • 最大序列数限制的是 batch 中有多少条活动请求，主要影响 decode 和一些按序列组织的元数据
  • 最大 token 数限制的是本轮总计算量，尤其影响 prefill 的算力和显存带宽压力

如果只按序列数限制，可能出现少量超长 prompt 把一次 prefill 打爆；如果只按 token 数限制，又可能在 decode 场景下塞进过多活动序列，导致调度和元数据膨胀。

我这里分别用了 max_num_seqs_ 和 max_num_batched_tokens_ 两个阈值。

这两个阈值你会静态配置，还是按运行时压力动态调整？

二、调度策略（确认你理解 schedule / preempt / postprocess 的控制流）

6 你会怎么描述一轮 schedule 的主流程？ 工程 ▶
面试官意图：考察你能不能清楚讲出调度循环，而不是只记住几个接口名
参考答案

我会把一轮调度分成两个阶段：

  1. 先从 waiting 队列头部开始尝试 prefill，检查 token 上限和 can_allocate()
  2. 如果本轮成功选出任意 prefill 序列，直接返回这些序列和 is_prefill=true
  3. 如果没有可做的 prefill，再进入 decode，从 running 队列头部取序列
  4. decode 前先检查 can_append()，不够就从 running 队尾挑受害者做抢占
  5. 可以继续时调用 may_append()，把该序列纳入本轮 decode batch

最终返回的是 (scheduled_seqs, is_prefill)，由 ModelRunner 决定按哪条执行路径准备输入。

你这里为什么选择“先完整尝试 prefill，再完整尝试 decode”，而不是混合批次？
7 prefill 计入 batch token 数时，为什么要扣掉 prefix cache 已命中的 token？ GPU ▶
面试官意图：确认你理解 prefix caching 改变的是“本轮实际计算量”，不是逻辑序列长度
参考答案

因为 batch token 上限要约束的是本轮真正要送进模型计算的 token 数，而不是序列逻辑长度。

如果前缀已经命中缓存，这部分 KV 已经存在，不需要重新 prefill。所以我的实现里在 prefill 路径中累加的是：

num_batched_tokens += seq->len() - seq->num_cached_tokens

这样调度器对计算量的估计才接近真实成本，也能让 prefix cache 真正转化成吞吐收益。

如果只扣整 block 命中的 token，而最后一个不完整 block 不扣，你觉得这个估算偏差大吗？
8 decode 阶段为什么不是每次都要分配新 block？什么时候才真的会新增 KV 页？ GPU ▶
面试官意图：检查你是否理解 block append 的边界条件
参考答案

因为一个 block 里可以容纳一段连续 token。decode 每轮虽然追加 1 个 token，但只有当前 token 跨入新 block 时才需要新分配。

在这个实现里，是否需要新 block 取决于 len() % BLOCK_SIZE。如果新 token 让位置落到一个新的 block 起点，才真正需要从 free list 里拿一个 block；否则只是继续写当前 block。

这也是为什么 decode 前不是直接分配，而是先调用 can_append() 评估“本轮可能需要的新 block”。

为什么 may_append() 的名字不是 append_block()？这个命名反映了什么语义？
9 当 decode 时发现显存页不够，为什么系统会选择抢占，而不是简单阻塞等待？ 工程 ▶
面试官意图：考察你对在线服务里阻塞、回退和资源让渡的理解
参考答案

因为在线推理里“阻塞等待资源”通常没有明确的释放时点。一个序列不往前跑，就不会产生 EOS，也就不会自然释放自己的 block，系统可能陷入整体停滞。

抢占的本质是：主动让一部分运行中的序列退回 waiting，回收它们占用的 block，让另一部分序列继续推进。这样系统至少能持续前进。

我当前实现的 preempt() 会把序列状态设回 WAITING，调用 block_manager_.deallocate() 释放 block，然后把它重新插回 waiting 队首。

被抢占后重新 prefill 的计算浪费，你会怎么向面试官解释这是可以接受的？
10 为什么这里选择 LIFO 抢占，也就是优先抢占最近加入 running 的序列？ 工程 ▶
面试官意图：看你是否能讲清策略背后的直觉，而不只是照着代码复述
参考答案

LIFO 的直觉是：越早进入 running 的序列，通常已经跑了更多轮，离完成更近。优先保住它们，系统更容易尽快释放一批资源。

相反，最近加入 running 的序列往往推进得更少，被抢占的沉没成本更低。所以 decode 阶段资源不足时，我是从 running_ 队尾拿受害者，而不是从队头拿。

这也是测试里专门覆盖的行为：当 block 不够时，被抢占的是最新进入 running 的那个序列。

有没有场景下 FIFO 抢占或者按剩余长度抢占会更合理？
11 被抢占的序列为什么要回到 waiting，而不是留在 running 里再打个 paused 标记？ 工程 ▶
面试官意图：考察你对状态机简化和后续重调度路径的理解
参考答案

回到 waiting 的好处是统一重入路径。一旦 block 被释放，这条序列下次再回来时，本质上就要重新走一次“能否重新拿到资源并被调度”的流程。

如果继续留在 running 再额外挂一个 paused 状态，调度器后面就得区分：

  • 真正可 decode 的 running
  • 资源已经被收走、但逻辑上还算 running 的 paused

这样队列语义会复杂很多。当前实现直接把它退回 WAITING，队列含义更干净。

如果以后支持 swap-out 到 CPU 内存，你还会让它回 waiting 吗？
12 postprocess 为什么属于调度器职责，而不是模型执行器直接改序列状态？ 工程 ▶
面试官意图：确认你知道执行平面和控制平面的职责边界
参考答案

因为 ModelRunner 负责的是给定一批序列，执行模型并产出 token；而“这个 token 产出之后序列是否结束、是否要回收 block、是否还留在 running”属于控制平面逻辑。

所以我把这些收口到 postprocess()：

  • 把新 token append 到序列
  • 检查 EOS / max_tokens
  • 若结束则标记 FINISHED 并释放 block
  • 把已结束序列从 running 队列移除

这样 ModelRunner 不需要关心队列和状态迁移，模块边界更清晰。

如果后面需要记录吞吐、每轮时延、完成原因，你会把统计打点放在 postprocess 还是更上层？

三、工程实现（确认你真正做过 C++ 版落地，而不只是复述 Python 伪代码）

13 如果在 C++ 里既想把序列放进容器，又想在队列里保存它们的指针，最容易踩的坑是什么？ C++ ▶
面试官意图：判断你是否真正考虑过对象生命周期和指针稳定性
参考答案

最大的坑是容器扩容或元素移动导致指针失效。如果一边在队列里存 Sequence*，一边又把对象本体放在会搬家的容器里，后面很容易悬空。

架构文档里最初是更简化的草图，但我实际落地时没有直接用“拥有对象的 deque 再配指针队列”，而是额外引入了独立所有权容器。

除了悬空指针，这种设计还可能带来哪些隐藏问题，比如 erase、重排或异常安全？
14 你在 C++ 版里是怎么保证调度队列里的序列指针稳定的？ C++ ▶
面试官意图：这题直接看你有没有真正实现过，而不是只知道理论流程
参考答案

我最后采用的是：

  • 用 std::list<Sequence> sequences_ 作为拥有对象的主容器
  • 用 std::deque<Sequence*> 维护 waiting_ 和 running_

新请求通过 add(Sequence) move 进 sequences_，然后把 &sequences_.back() 放进 waiting 队列。因为 std::list 节点地址稳定，所以队列里的指针可以长期有效。

这是一个典型的 C++ 工程化修正：逻辑上遵循原始调度算法，但实现上补足对象生命周期安全。

如果后面需要彻底删除 FINISHED 的序列，你会在什么时机从 sequences_ 擦除？
15 调度器里“结束条件”的判断通常有哪些坑？ LLM ▶
面试官意图：看你是否考虑过 EOS、ignore_eos 和 max_tokens 之间的组合关系
参考答案

常见坑有三个：

  • EOS 判断和 ignore_eos 冲突：如果用户要求忽略 EOS，就不能提前停
  • max_tokens 是 completion 长度，不是总长度
  • 先 append 再判断：因为当前轮产出的 token 也属于生成结果

我在 postprocess() 里是先 append_token()，再检查：

(!seq->ignore_eos && token_id == eos_) ||
seq->num_completion_tokens() == seq->max_tokens

满足条件就标记 FINISHED 并回收 block。

如果支持 stop words、多 token 终止串，结束判断会落在哪一层？
16 你会怎么验证这个调度器实现是对的？至少要覆盖哪些测试场景？ 工程 ▶
面试官意图：考察你是否真的把设计落成了可验证的代码
参考答案

我会优先覆盖三类场景，这也是我当前测试文件里的主线：

  • Prefill 优先：已有 decode 序列时，新请求进入后应优先返回 prefill batch
  • 显存不足触发抢占：decode 阶段 block 不够时，最新 running 序列应被抢占回 waiting
  • postprocess 正确结束并回收资源：命中 EOS 后状态变 FINISHED，block_table 清空，资源可被后续序列复用

除了这些，我还会补边界测试，比如空队列、只能抢占自己、prefix cache 命中后 batched token 统计是否正确等。

如果线上出现“偶发性卡死”，你第一反应会加哪类断言或日志？
17 如果后面支持 speculative decoding，这个 Scheduler 的接口大概率要怎么演进？ LLM ▶
面试官意图：看你能否从当前实现出发，推演下一阶段系统演进方向
参考答案

现在的接口默认每轮每个序列只确认 1 个 token，所以 postprocess() 直接接收一组标量 token。

如果支持 speculative decoding，一轮里同一个序列可能确认多个 token，那至少要变两点：

  • 执行结果表示：从“每序列一个 token”变成“每序列一段已确认 token”
  • 资源维护：一次 postprocess 中可能跨越多个 block 边界，可能多次触发 append 逻辑

届时更合适的接口可能是按序列返回一个结构体，里面包含 confirmed tokens、是否回退、以及额外统计信息，而不是只传 vector<int32_t>。

你觉得 speculative decoding 更应该改动 Scheduler，还是更应该改动 ModelRunner 和 postprocess 协议？
18 这个调度器现在最明显的局限是什么？如果让你继续演进，你会先做哪一步？ 工程 ▶
面试官意图：看你是否能客观评价当前实现，而不是把它包装成“最终最优解”
参考答案

当前实现的优点是清晰、可验证，但局限也很明显：

  • prefill 优先可能导致 decode 饥饿
  • 抢占策略只有 LIFO，一刀切，缺少基于代价的选择
  • 没有 chunked prefill，也没有分层配额
  • postprocess 只支持每轮 1 token 的简单协议

如果继续演进，我会优先做两件事之一：

  1. 加入更细粒度的公平性策略，比如给 decode 保留最低配额
  2. 把 batch 形成逻辑从“二选一”扩展到更接近线上系统的混合策略

这样既能保留现在的清晰控制流，也能更贴近真实服务场景。

如果只能选一个指标优化，你会优先首 token latency、tokens/s，还是 P99？
