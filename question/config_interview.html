1 LLM 推理引擎的显存一般是怎么划分的？你做配置的时候怎么考虑显存这块？ GPU ▶
面试官意图：考察对 GPU 显存布局的全局理解
参考答案

LLM 推理引擎的显存主要分三块：

  • 模型权重（固定占用）— 加载后不变，如 Qwen3-0.6B bf16 约 1.2GB
  • 中间激活（随 batch 变化）— prefill 大 batch 时临时占用可能很大
  • KV Cache（动态增长）— 每个请求每生成一个 token 就增长一点，是最大的变量

我在配置里加了一个显存利用率参数，默认 0.9，含义是引擎最多使用 90% 的显存来预分配 KV Cache pool。留 10% 余量给中间激活、CUDA context、driver 开销等。不设 100% 是因为：prefill 大 batch 时激活临时占用可能很大、cudaMalloc 存在碎片、同卡上可能还有其他进程。

这个比例是针对 KV Cache 的，还是全局显存预算？你怎么从这个比例算出能分配多少 KV Cache block？
2 Transformer 模型的上下文长度受什么约束？你的配置模块怎么处理上下文长度这件事？ LLM ▶
面试官意图：区分模型能力上界 vs 运行时资源约束
参考答案

上下文长度受两层约束：

1. 模型层面：训练时位置编码（如 RoPE）预计算到一个上限，超过这个长度位置编码就失效了。这是模型的能力天花板。

2. 硬件层面：KV Cache 占用与序列长度成正比。A100 80GB 能支撑 32K 上下文，消费显卡可能只能支撑 2K。

我的做法是分成两个配置：从模型 config.json 读出模型的位置编码上限（只读，不可改），用户另外指定运行时的上下文长度限制。启动时校验：如果用户设的值超过模型上限，自动截断并输出警告。这样同一个模型在不同硬件上可以用不同的上下文长度，模型文件不用改。

如果用户设的值超过模型上限，你是选择报错还是静默截断？为什么这样选择？
3 说说你对多卡推理的理解吧。张量并行、数据并行、流水线并行，在推理场景下怎么选？ GPU ▶
面试官意图：分布式推理基础认知
参考答案
  • 数据并行（DP）：每卡一份完整模型副本，不同卡处理不同请求。推理场景用得少——权重冗余太多，小模型一张卡就够，大模型一张卡放不下
  • 张量并行（TP）：一层的权重按维度切到多卡（如 attention heads 平分），每步需要 all_reduce。适合单机多卡、延迟敏感的场景
  • 流水线并行（PP）：不同层分到不同卡，按流水线传递。适合多机、吞吐优先的场景，但有 bubble 延迟

推理引擎通常选 TP，因为推理时 batch 可以很小（甚至 1 条请求），DP 没意义；PP 有 bubble；TP 能降低单条请求的延迟。我的配置里有一个 TP size 参数，限制为 1-8（单机最多 8 卡），影响后续权重加载时的切分方式和 attention head 的分配。

TP 的 all_reduce 通信开销大概在什么量级？什么情况下 TP=4 反而比 TP=2 慢？
4 推理引擎中 KV Cache 的内存管理一般怎么做？和传统的预分配方式有什么区别？ LLM ▶
面试官意图：PagedAttention 的理解
参考答案

传统做法是为每个请求按最大序列长度预分配 KV Cache 连续内存——一个只需 100 token 的请求也占满 4K 或更高的空间，显存利用率通常只有 ~50%。

现在主流方案是 PagedAttention，借鉴操作系统虚拟内存分页：把 KV Cache 切成固定大小的 block（页），按需分配。请求实际用了多少 token 就分配多少 block，利用率可以达到 ~95%。

我的配置里有一个 block size 参数，表示每页存储多少个 token 的 KV。设大了分配粒度粗（尾部浪费多）但管理开销小；设小了粒度细但 block table 更大、调度更频繁。我把它限制为 2 的幂且不小于 16，方便对齐 GPU 访存。

block size 选大还是选小有什么 trade-off？主流推理引擎一般用多大？
5 现在很多模型不再用标准的 Multi-Head Attention 了，你了解哪些变体？它们对推理引擎有什么影响？ LLM ▶
面试官意图：GQA/MQA 的理解，以及对 KV Cache 占用的量化分析
参考答案

主要三种：

  • MHA（Multi-Head Attention）：Q/K/V head 数量相同。经典但 KV Cache 开销最大
  • MQA（Multi-Query Attention）：所有 Q head 共享 1 组 KV。KV Cache 最小，但质量有损失
  • GQA（Grouped Query Attention）：Q head 分组，每组共享 1 组 KV。是 MHA 和 MQA 的折中，目前最主流（LLaMA 2 70B、Qwen3 都用）

对推理引擎的影响：KV Cache 大小直接取决于 KV head 数量。比如 GQA ratio = 2 时 KV Cache 就只有 MHA 的一半。配置模块需要分别读取 Q head 数和 KV head 数，这两个值不同直接影响 KV Cache pool 的大小计算和显存预算。

如果 KV head 只有 1 个，那就是 MQA 了——它有什么问题？为什么现在大家更倾向 GQA？
6 推理引擎通常会做一些 GPU 执行层面的优化来降低 kernel launch 开销，你了解吗？配置模块需要为此做什么？ GPU ▶
面试官意图：CUDA Graph 的工程理解
参考答案

主要是 CUDA Graph — 把一系列 GPU kernel 录制成一张图，重放时跳过 CPU 端逐个 launch 的开销（每个 launch ~10-20μs）。decode 阶段 kernel 计算量小但数量多，Graph 能提速 10-30%。

但 Graph 有限制：录制时 tensor 地址必须固定、不支持动态控制流、调试困难（不能断点/print）、capture 多个 batch size 组合会额外占显存。

所以我在配置里加了一个开关参数，默认启用 Graph，但调试时可以强制关闭回到逐个 launch 的 eager 模式。这个参数对运行时完全透明——配置层只管传递开关，真正的 capture/replay 逻辑在执行器层。

CUDA Graph capture 的时候为什么 input tensor 地址必须固定？不固定会怎样？

7 你的配置模块需要解析 HuggingFace 模型目录里的 JSON 文件，解析第三方格式你怎么保证健壮性？ 工程 ▶
面试官意图：边界输入处理和防御性编程

我的策略是"必需字段严格、可选字段宽松"：

  • 必需字段（如 hidden_size、num_layers）用严格取值，缺失立刻抛异常，错误信息明确指出哪个字段缺了
  • 可选字段（如 tie_word_embeddings）先检查存在性再用默认值。不同模型架构的 config.json 字段不完全一致

HuggingFace 的 config.json 格式非常稳定，但每个新架构可能加新字段。"必需严格、新字段宽松"的策略很适合这个场景。

8 配置模块在整个推理引擎架构中是什么角色？它和调度器、模型执行器这些模块怎么交互？ 工程 ▶
面试官意图：全局架构感知和生命周期管理
参考答案

Config 是整个引擎最先创建、全程只读的对象：

程序启动
  → 解析配置（第一步）
  → 调度器拿到配置 → 确定 batch 策略、队列上限
  → 模型执行器拿到配置 → 确定显存分配、模型结构
  → 引擎主类组装所有组件
  → 推理循环……
  → 程序退出

交互方式：const 引用传递。Config 创建后不会被修改，所有下游模块只读使用。唯一的例外是 KV Cache block 数量——这个值需要模型加载后才能计算出来（要知道模型权重占了多少显存才能算剩余空间），所以在配置里是延迟填充的。

这种"大部分字段创建时确定，个别字段延迟填充"的设计你觉得好吗？有什么更好的方案？
9 C++ 项目中做 JSON 解析，有不少库可以选，你是怎么考虑选型的？ C++ ▶
面试官意图：技术选型的思考过程和 trade-off 分析
参考答案

主流三个库定位不同：

  • nlohmann/json：API 最直觉（像 Python dict），header-only 零配置，STL 兼容好。但解析速度最慢、编译时间长
  • rapidjson：性能好 5-10 倍，header-only，DOM 和 SAX 两种模式。但 API 繁琐（手动 HasMember/GetInt）
  • simdjson：SIMD 加速，单核 GB/s 级解析速度。但只读、需要编译库

我选了 nlohmann/json，原因：Config 解析是启动时一次性操作，config.json 只有 ~30 行，性能完全不是瓶颈。开发效率远大于运行效率。如果场景是解析 GB 级日志或 streaming JSON，那 simdjson 或 rapidjson 才有意义。

header-only 库如果很多 .cpp 都 include 了它，编译时间会怎样？你怎么控制？

11 你的配置里既有模型结构信息（层数、维度这些），也有运行时可调参数（并发数、显存比例），你怎么组织这些数据？ 工程 ▶
面试官意图：数据建模能力，分层设计
参考答案

我把它们分成两个 struct：

  • 模型配置（来自 config.json）：hidden_size、num_layers、num_heads、vocab_size 等。这些是模型训练者定义的固有属性，用户不应改也不能改。同一个模型在任何机器上这些值都一样
  • 引擎配置（来自用户/运维）：上下文长度、batch 上限、显存比例、TP size 等。这些根据硬件和业务需求调整

引擎配置内部包含模型配置作为成员。好处是职责清晰：拿到引擎配置就能访问一切，但模型配置是只读子结构，改不了。

12 配置模块的职责边界在哪？它只管解析配置，还是也做别的事情？ 工程 ▶
面试官意图：单一职责原则的实践理解
参考答案

Config 模块只负责解析配置文件 + 校验参数 + 推导衍生值。不做任何资源加载：

  • 做的：读 JSON → 填 struct → 校验合法性 → 推导衍生值（如 head_dim = hidden_size / num_heads）
  • 不做的：加载模型权重、初始化 tokenizer、分配 GPU 显存、创建 CUDA context

设计原则：Config 是纯 CPU、无 GPU 依赖、无副作用的模块（唯一 I/O 是读一个小 JSON）。好处是可以独立编译、独立测试、不需要 GPU 环境。如果把权重加载也塞进来，测试就需要整个 GPU 环境，变成了集成测试。

如果后面加一个需求：启动时自动检测 GPU 型号和显存大小来设默认值，你还觉得它应该"无 GPU 依赖"吗？怎么处理这个矛盾？

三、实战与扩展（开放题，考察经验广度和思考深度）

13 如果要支持不同精度的推理——FP16、BF16、INT8 甚至 FP8——配置层需要怎么考虑？ LLM ▶
面试官意图：量化/混合精度推理的认知和配置扩展能力
参考答案

模型 config.json 里有训练时的精度，但推理精度不一定跟它一样。需要加：

  • 推理精度：一个 dtype 字段，"auto" 表示跟训练精度一致，也可以手动指定
  • 量化配置：INT8/INT4 推理需要量化方法（AWQ / GPTQ / SmoothQuant）、量化位数、group size 等
  • KV Cache 精度：KV Cache 可以独立做 FP8/INT8 压缩，和权重精度是独立的两个维度

BF16 vs FP16：BF16 指数位同 FP32，范围大不易溢出，适合 attention logits；FP16 精度高但范围小。Ampere+ 硬件两者 Tensor Core 性能一样。精度选择直接影响显存预算——INT8 权重比 FP16 小一倍，能省出空间给更大 KV Cache pool。

INT8 量化的 KV Cache 对推理质量影响大吗？什么场景下值得做？

16 如果要支持多种模型架构——不只一种——你的配置模块怎么兼容？ 工程 ▶
面试官意图：扩展性设计思维
参考答案

好消息是大部分 Transformer 解码器模型的 config.json 字段高度重叠：hidden_size、num_heads、num_layers、vocab_size 几乎通用。差异在于：

  • 激活函数：SiLU（SwiGLU）vs GELU

17 做推理引擎和做训练框架，在配置设计上有什么本质区别？ LLM ▶
面试官意图：训练 vs 推理的认知广度
参考答案

训练配置关注：学习率/optimizer/scheduler、梯度累积步数、混合精度策略（loss scaling）、数据集路径、checkpoint 保存策略、DP+TP+PP+ZeRO 的复杂并行组合

推理配置关注：显存分配策略（KV Cache 预算）、批处理参数（最大并发数、prefill token 预算）、延迟优化开关（CUDA Graph、投机解码）、量化配置

本质区别：训练追求吞吐和收敛，配置围绕计算和通信；推理追求延迟和并发，配置围绕内存和调度。训练的显存瓶颈在激活和梯度（activation checkpointing 换空间），推理的显存瓶颈在 KV Cache（分页换利用率）。

18 最后问一个开放题：回顾你实现的这个配置模块，你觉得有什么不足？如果重新做一遍会改什么？ 工程 ▶
面试官意图：自省能力和工程成熟度
参考答案

可以坦诚反思的点：

  • 参数来源单一 — 只支持 JSON 文件，实际应该支持环境变量覆盖（方便容器化部署）和命令行参数
  • 缺少日志 — 自动修正参数时（如截断上下文长度）没有 warning 日志，用户可能不知道值被改了
  • 延迟初始化不够优雅 — 有个字段初始值是 -1 表示"还没算"，应该用 std::optional 更类型安全

  • 没有版本兼容机制 — 如果以后配置格式变了，没有 schema version 来做向前兼容
  • 缺少 dump 功能 — 上线后排查问题需要看当前生效的完整配置，应该能序列化回 JSON

关键是说出不足的同时给出改进方案，而不是只说"挺好的没问题"。面试官看的是你有没有踩过坑、有没有真正思考过工程质量。

四、Attention 机制（基础原理）

19 从头到尾描述一下 Self-Attention 的完整计算流程。 LLM ▶
面试官意图：确认候选人不是只知道"Q K V"三个字母，而是理解完整的数据流
参考答案

给定输入 X（shape: [seq_len, hidden_size]），完整流程：

  1. 线性投影：Q = X·W_q，K = X·W_k，V = X·W_v。三个投影矩阵 shape 均为 [hidden_size, hidden_size]
  2. 分头（Reshape）：将 Q/K/V 从 [seq_len, hidden_size] reshape 为 [num_heads, seq_len, head_dim]
  3. 缩放点积：scores = Q · K^T / √d_k，shape: [num_heads, seq_len, seq_len]
  4. Mask（可选）：causal mask 把上三角设为 -∞，防止看到未来 token
  5. Softmax：对最后一个维度做 softmax，得到注意力权重（每行加和为 1）
  6. 加权求和：output = softmax(scores) · V，shape: [num_heads, seq_len, head_dim]
  7. 合并头（Concat）：reshape 回 [seq_len, hidden_size]
  8. 输出投影：output = output · W_o，最终 shape: [seq_len, hidden_size]

20 Attention 计算中为什么要除以 √d_k？不除会怎样？ 数学 ▶
面试官意图：考察对数值稳定性的数学理解
参考答案

假设 Q 和 K 的每个元素独立、均值 0、方差 1，那么 Q·K^T 中每个元素是 d_k 个乘积的求和：

q · k = Σ(q_i · k_i)，其中每个 q_i·k_i 的方差 = 1，求和后方差 = d_k

即 点积结果的方差随 d_k 线性增长。d_k = 128 时，点积值的标准差约 11.3。

不除的后果：

  • softmax 饱和：输入值很大时，softmax 输出趋近 one-hot（一个接近 1，其余接近 0）
  • 梯度消失：softmax 在饱和区的梯度接近 0，反向传播梯度几乎不流动
  • FP16 溢出：FP16 最大值 65504，大点积可能直接变 inf

除以 √d_k 后，方差归一到 1，softmax 工作在合理区间，梯度正常传播。

如果用 BF16 推理，softmax 的数值稳定性还有什么额外需要注意的？（提示：BF16 尾数只有 7 位）
21 Multi-Head Attention 的"多头"到底有什么用？为什么不用一个大 head？ LLM ▶
面试官意图：理解多子空间建模的直觉和工程意义
参考答案

理论角度：

  • 每个 head 在独立的低维子空间中学习不同的注意力模式——有的 head 学句法（主语关注谓语），有的学指代（代词关注先行词），有的学局部上下文
  • 一个大 head 会把所有模式混在一起，softmax 只能输出一个注意力分布，无法同时关注多个不同位置的不同语义

工程角度：

  • 多 head 的 Q·K^T 矩阵更小（[seq, head_dim] 而非 [seq, hidden_size]），计算效率更高
  • 多 head 天然适合 GPU 并行——每个 head 独立计算，可以并行在不同 CUDA thread block 上
  • 张量并行（TP）时按 head 切分——正好一组完整的 head 分到一张卡

数值直觉：hidden_size=1024，num_heads=16 → 每个 head 处理 64 维。16 个 64 维的注意力模式比 1 个 1024 维的更灵活。

22 MHA、MQA、GQA 三种注意力变体的原理和区别是什么？在推理引擎里它们各自的影响？ LLM ▶
面试官意图：GQA 是当前主流配置，必须能清晰区分三者
参考答案

三者的核心区别在于 KV head 的数量：

  • MHA：num_kv_heads = num_q_heads。每个 Q head 有独立的 KV head。KV Cache 最大
  • MQA：num_kv_heads = 1。所有 Q head 共享同一组 KV。KV Cache 最小，但表达能力受限
  • GQA：1 < num_kv_heads < num_q_heads。Q head 分组，每组共享一组 KV。折中方案

KV Cache 对比（以 32 层、hidden_size=4096、head_dim=128、BF16、seq_len=4K 为例）：

每 token 每层 KV Cache = 2 × num_kv_heads × head_dim × 2 bytes

MHA (32 kv_heads): 2 × 32 × 128 × 2 = 16,384 bytes
GQA (8 kv_heads):  2 ×  8 × 128 × 2 =  4,096 bytes  ← 4倍节省
MQA (1 kv_head):   2 ×  1 × 128 × 2 =    512 bytes  ← 32倍节省

GQA 成为主流（LLaMA2-70B、Qwen3、Mistral），因为 MQA 虽然更省但质量下降明显（KV 信息瓶颈），GQA 在显存和质量之间取得了最佳平衡。

23 Causal Mask 是什么？为什么 decoder-only 模型一定需要它？ LLM ▶
面试官意图：理解自回归生成的基础约束
参考答案

Causal Mask 是一个下三角矩阵，上三角部分填 -∞（softmax 后变为 0）。效果：位置 i 只能 attend 到位置 ≤ i 的 token。

token:  [A, B, C, D]

Mask:   A  B  C  D
    A [ 0 -∞ -∞ -∞ ]
    B [ 0  0 -∞ -∞ ]
    C [ 0  0  0 -∞ ]
    D [ 0  0  0  0 ]

为什么必须有：

  • 训练时，一个序列的所有位置并行计算。如果不 mask，位置 2 的 token 在训练时能看到位置 3、4 的 token → 训练和推理不一致（推理时生成位置 2 时位置 3 还不存在）
  • 有了 causal mask，训练中每个位置只看到"过去"→ 和推理时的自回归行为一致

推理时的工程影响：

  • Prefill 阶段：需要完整 causal mask（是一个 [seq_len, seq_len] 的矩阵）
  • Decode 阶段：每次只新增 1 行，且自然看不到未来（因为未来还没生成），所以 mask 退化为"无需 mask"——Flash Attention 的 causal 模式会自动处理

24 说说 Flash Attention 的核心思想，它解决了什么问题？ GPU ▶
面试官意图：理解 IO-aware 算法设计思想
参考答案

问题：标准 Attention 计算 Q·K^T 产生一个 [seq_len, seq_len] 的矩阵，seq_len=4K 时要 64MB 显存（FP16），32K 时要 4GB。这个中间矩阵必须存到 HBM（高带宽显存），大量 HBM 读写成为瓶颈。

Flash Attention 的核心：分块（Tiling）+ Online Softmax

  1. 分块：不一次算整个 [seq, seq] 矩阵，而是把 Q 分成小块（fit in SRAM/Shared Memory，~192KB），每次只算一块 Q 和全部 K 的注意力
  2. Online Softmax：softmax 通常需要整行的全局 max 和 sum。Flash Attention 用 online 算法，流式更新 max 和 sum，不需要完整存储中间矩阵
  3. 不 materialize 中间矩阵：scores 矩阵在 SRAM 中计算、softmax、和 V 相乘后直接丢弃，不写回 HBM

效果：

  • 显存从 O(n²) 降到 O(n)——不需要存 [seq, seq] 矩阵
  • 速度提升 2-4x——减少了大量 HBM 读写（GPU 计算快，IO 慢）
  • 支持更长序列——显存不再是瓶颈
Flash Attention 的反向传播怎么做？不存中间矩阵的话，梯度怎么算？（提示：recomputation）

25 Transformer 为什么需要位置编码？不加会怎样？ LLM ▶
面试官意图：理解 Transformer 的置换不变性（permutation invariance）
参考答案

Self-Attention 的计算对输入序列的顺序完全不敏感——把 token 打乱后，每个 token 的 attention 权重和输出完全一样（只是行列重排了）。这称为置换不变性（permutation invariance）。

数学证明：对任意置换矩阵 P，Attention(PX) = P · Attention(X)。注意力结果只取决于 token 之间的"相似度"，和它们在序列中的位置无关。

不加位置编码的后果：

  • "猫吃鱼" 和 "鱼吃猫" 对模型完全等价——无法区分语序
  • 模型退化为一个无序的 token 集合处理器（bag of tokens）
  • 对于需要理解顺序的任务（语言生成、翻译、代码）完全失效

位置编码的作用：给每个 token 注入其在序列中的位置信息，打破置换不变性。

26 说说原始 Transformer 的正弦位置编码的数学原理，它为什么能编码位置？ 数学 ▶
面试官意图：数学推导能力，频率分解思想
参考答案

原始公式（位置 pos，维度 i）：

PE(pos, 2i)   = sin(pos / 10000^(2i/d))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d))

数学直觉：把每个位置编码为不同频率的正弦波的叠加——低维对应高频（变化快，区分相邻位置），高维对应低频（变化慢，编码全局位置）。

关键性质 —— 平移不变性：

对于固定偏移 k，存在一个线性变换 T_k 使得 PE(pos+k) = T_k · PE(pos)。这意味着模型可以通过学习 T_k 来捕捉"相对距离为 k"这个概念，不需要为每个具体位置单独学习。

证明核心：sin(α+β) = sinα·cosβ + cosα·sinβ，所以 PE(pos+k) 可以表示为 PE(pos) 的线性组合，系数只取决于 k。

10000 这个 base：让最高频维度的波长 = 2π（≈6.28 个位置一个周期），最低频维度的波长 = 10000·2π（≈62832 个位置一周期）。覆盖了从短距离到长距离的范围。

27 RoPE（旋转位置编码）的完整原理是什么？为什么现在大多数 LLM 都用它？ 数学 ▶
面试官意图：RoPE 是当前 LLM 标配，必须能清晰推导
参考答案

核心思想：对 Q 和 K 向量施加一个与位置相关的旋转，使得 Q·K^T 的结果只取决于相对位置。

推导过程：

1. 把 head_dim 维向量的每两个相邻维度看作一个 2D 复数平面

2. 位置 m 处，第 i 对维度旋转角度 θ = m · ω_i，其中频率 ω_i = 1 / (base^(2i/d))

3. 用旋转矩阵表示：

R(m, i) = [cos(m·ω_i)  -sin(m·ω_i)]
           [sin(m·ω_i)   cos(m·ω_i)]

4. Q' = R(m) · Q，K' = R(n) · K

5. 关键性质：Q'^T · K' = Q^T · R(m)^T · R(n) · K = Q^T · R(n-m) · K

旋转矩阵是正交的，R(m)^T · R(n) = R(n-m)，所以点积只依赖相对位置 n-m。

为什么 RoPE 成为标配：

  • 天然编码相对位置（不需要额外参数）
  • 可以推广到训练时没见过的长度（配合 θ 调整）
  • 计算开销极小——每个 token 的 Q/K 只需做一次逐元素的 sin/cos 乘法
  • 和 Flash Attention 等优化完全兼容
RoPE 的 base（theta）对上下文长度有什么影响？为什么 LLaMA 用 10000 而 Qwen3 用 1000000？
28 RoPE 的 base（theta）值大小对模型有什么影响？长上下文模型是怎么扩展的？ LLM ▶
面试官意图：理解 RoPE 外推 / NTK scaling 的工程意义
参考答案

频率公式：ω_i = 1 / (theta^(2i/d))

theta 的影响：

  • theta 越大 → 所有 ω_i 越小 → 旋转越慢 → 相邻位置的角度差越小
  • 好处：高维度在更长的序列范围内仍有变化 → 支持更长上下文
  • 代价：区分相邻位置的能力降低（角度差太小时难以区分 token 502 和 503）

长上下文扩展方法：

  • 直接提高 theta（如 10000 → 1000000）：最简单，但可能需要 fine-tune 适配
  • NTK-aware scaling：非均匀调整——低频维度（编码长距离的）放大更多，高频维度（编码短距离的）保持不变。类比：调望远镜的焦距但不影响近景清晰度
  • YaRN：在 NTK 基础上加 attention scaling factor，进一步减少外推时的 perplexity 退化
  • Linear scaling：所有 ω 除以比例因子。最简单但质量最差——高频信息被压缩

数值直觉：theta=10000 时，head_dim=128 最高维度的半周期约 5000 个 token。theta=1000000 时，约 500000 个 token。

如果模型训练时用 theta=10000、max_len=4K，推理时直接用 32K 不做任何调整，效果会怎样？perplexity 曲线是什么形状？
29 除了 RoPE，还有哪些位置编码方案？各自的优缺点？ LLM ▶
面试官意图：考察技术视野的广度
参考答案

1. 绝对位置编码：

  • Sinusoidal PE（原始 Transformer）：固定公式生成，无额外参数。但不编码相对位置
  • Learned PE（GPT-2/BERT）：可学习的 embedding lookup table。简单有效，但长度外推差——训练 512 token 就无法处理 513

2. 相对位置编码：

  • RoPE（LLaMA/Qwen/Mistral）：旋转实现相对编码，计算高效，外推好。当前 decoder-only 标配
  • ALiBi（BLOOM/MPT）：不改 Q/K，而是在 attention score 上加一个与距离成比例的线性 bias。零额外参数、外推极好，但表达能力稍弱
  • T5 bias：学习一个离散的 relative position bias table。表达能力好但不能外推

对推理引擎的影响：

  • RoPE：在每步 decode 时对 Q/K 做 sin/cos 乘法，非常快
  • ALiBi：在 attention score 上加 bias，需要额外的 bias tensor（但不占 KV Cache）
  • Learned PE：推理时只是一次 embedding lookup，最简单

30 RoPE 在 GPU 上是怎么实现的？它的计算开销如何？ GPU ▶
面试官意图：从数学到工程落地的能力
参考答案

朴素实现：构造 2x2 旋转矩阵做矩阵乘。但 2x2 矩阵乘太小，利用不了 Tensor Core。

实际实现（逐元素）：利用旋转矩阵的特殊结构，展开为：

q[2i]'   = q[2i] * cos(θ) - q[2i+1] * sin(θ)
q[2i+1]' = q[2i] * sin(θ) + q[2i+1] * cos(θ)

只需 4 次乘法 + 2 次加法 per 2 个元素。是逐元素操作，完美并行。

工程优化：

  • 预计算 cos/sin table：启动时预计算所有位置的 cos/sin 值存为 tensor（shape: [max_len, head_dim/2]），推理时 lookup 即可
  • 融合到 QKV 投影 kernel：不单独起一个 RoPE kernel，而是在 attention kernel 内部做。减少一次 HBM 读写
  • half2 向量化：BF16/FP16 下用 half2 做 2 个元素并行运算

开销极低：对于一个 decode step（batch=1, seq=1），RoPE 只处理 num_heads × head_dim 个元素的 sin/cos 乘法，远小于 attention 的 GEMM 开销（通常 <1% 总时间）。

32 RMSNorm 和 LayerNorm 有什么区别？为什么现在的 LLM 几乎都用 RMSNorm？ 数学 ▶
面试官意图：归一化方法间的数学和工程 trade-off
参考答案

LayerNorm：

y = (x - mean(x)) / √(var(x) + ε) × γ + β

需要计算 mean 和 var（两次 reduce），有两组可学习参数（γ scale + β bias）。

RMSNorm：

y = x / √(mean(x²) + ε) × γ

只计算 x² 的均值（一次 reduce），只有 γ scale 参数，没有 β bias。

为什么 LLM 偏好 RMSNorm：

  • 计算量少 ~30%：省掉 mean 和 bias。每层执行两次 norm，28 层就是 56 次——累积起来不小
  • 参数量少：没有 bias，每层少 hidden_size 个参数
  • 实验效果相当：RMSNorm 论文证明 re-centering（减 mean）对 Transformer 效果影响极小，re-scaling（除 RMS）才是关键
  • CUDA kernel 更简单：一次 reduce 比两次 reduce 更容易做到 warp-level 高效实现

33 SwiGLU 是什么？为什么现在的 LLM 的 FFN 层都用它而不是普通的 ReLU + 两层 MLP？ LLM ▶
面试官意图：FFN 层的演进和 GLU 变体的理解
参考答案

传统 FFN：

FFN(x) = ReLU(x · W₁) · W₂

SwiGLU FFN：

FFN(x) = [SiLU(x · W_gate) ⊙ (x · W_up)] · W_down

其中 SiLU(x) = x · σ(x)（Swish 激活），⊙ 是逐元素相乘。

GLU 的核心思想：引入一个门控机制——W_gate 控制哪些特征通过（门的开/关由输入内容决定），W_up 做实际的特征变换。相比简单的 ReLU（硬门控：<0 全截断），SiLU 是平滑的门控，梯度流更好。

代价：多了一个 W_gate 矩阵。为了总参数量不变，intermediate_size 从 4h 缩小到 ~2.7h（8h/3 取整）。所以看到 intermediate_size=3072 而非 4096（对于 hidden_size=1024）。

工程影响：

  • FFN 从 2 个大 GEMM 变成 3 个 GEMM（gate + up + down）
  • gate 和 up 可以融合为一个 GEMM（concat 后一起算），实际只是 2 个 GEMM
  • SiLU 是逐元素操作，绑在逐元素乘法 kernel 里几乎零开销

位置编码：
